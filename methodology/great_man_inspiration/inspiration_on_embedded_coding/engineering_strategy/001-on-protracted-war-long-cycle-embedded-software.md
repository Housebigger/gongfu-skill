# 从《论持久战》看长周期嵌入式软件工程的方法论启示

日期：2026-04-14

原文链接：
https://www.marxists.org/chinese/maozedong/marxist.org-chinese-mao-193805b.htm

说明：
本文不是把战争语言直接照搬到工程管理里，而是试图抽取《论持久战》中的方法论结构，讨论它对长周期嵌入式软件工程，尤其是嵌入式编码、调试、联调、量产收敛与长期维护工作的现实启示。

## 一、为什么《论持久战》值得工程人重读

《论持久战》最强的地方，不只是它在特定历史条件下对战争进程的判断，更在于它呈现了一种处理复杂长期问题的方法：

- 不被一时强弱吓倒
- 不被短期胜利冲昏
- 不把局部突破误当全局完成
- 不用情绪代替分析
- 不用愿望代替阶段判断
- 不把长期任务做成无计划的消耗

这套方法，如果抽离战争背景，放进长周期嵌入式软件工程里，竟然非常贴切。

因为嵌入式项目天然具备几个特点：
- 依赖复杂：硬件、驱动、工具链、协议栈、业务逻辑、生产环境相互耦合
- 周期长：从 bring-up 到量产再到现场维护，远长于普通应用层开发
- 反馈慢：很多问题只有在真机、真环境、长时间运行或批量出货后才暴露
- 资源硬约束强：CPU、内存、Flash、功耗、总线时序、实时性都不是“多加点资源”就能解决的
- 技术债隐蔽：今天能跑，不代表三个月后可维护；实验室能过，不代表现场稳定

因此，嵌入式工程最怕两种错误：一种是“根本做不成”的悲观瘫痪，另一种是“很快就做完”的虚假乐观。

《论持久战》正好同时反对这两种东西。

## 二、工程里的“亡国论”和“速胜论”

### 1. 什么是嵌入式工程里的“亡国论”

常见说法包括：

- 这平台太老了，根本没法做
- 厂商 SDK 太差，项目注定崩
- 历史包袱太重，重构也没用
- 板卡问题这么多，不可能按期收敛
- 现场问题太随机，根本定位不了

这类判断的问题不在于它完全错，而在于它只看见困难，不看见动态条件变化。

《论持久战》的启发是：
不能只看“敌强我弱”这一面，还要看其它矛盾因素。

放到工程里，就是：
- 虽然硬件差，但需求可能可裁剪
- 虽然架构旧，但可先分层隔离
- 虽然工具链烂，但可补自动化脚手架
- 虽然现场复杂，但可增强日志、trace 和 dump
- 虽然团队薄，但可通过文档化和模块化提升组织能力

所以很多项目不是“不能做”，而是“不能急做、不能乱做、不能幻想一次做完”。

### 2. 什么是嵌入式工程里的“速胜论”

另一种极常见错误，是把局部突破误当全局胜利。

常见表现包括：

- 板子点亮了，后面就简单了
- 驱动跑通了，系统差不多了
- demo 已经出来了，量产问题不大
- 再加几个人，很快就能追上
- 先把功能堆上去，稳定性以后再说

这类想法的问题在于：
嵌入式系统的真正难点，很多根本不在“能不能跑”，而在：

- 能否稳定跑
- 能否长期跑
- 能否在边界条件下跑
- 能否批量一致地跑
- 能否出现故障后可诊断、可恢复
- 能否在团队人员变化后继续维护

因此，工程版《论持久战》的第一原则是：
既反对因困难而自我瘫痪，也反对因局部胜利而盲目冒进。

## 三、最重要的借鉴：用“阶段论”管理长周期项目

《论持久战》对“阶段”的强调，对工程项目极有启发。

长周期嵌入式软件工程，不应被理解成一个连续均匀推进的任务，而更像几个性质不同的阶段。

### 1. 第一阶段：生存期

这相当于工程的“战略防御期”。

重点不是完美，而是建立最小生存能力：

- 编译链可用
- 烧录链可用
- 启动链可用
- 基础日志可见
- 时钟、电源、存储、总线基础稳定
- Bootloader 和最小系统可启动
- 首批关键外设能工作

这一阶段的核心不是“大而全”，而是“先活下来”。

对嵌入式来说，所谓“活下来”就是：
- 系统能启动
- 状态能观察
- 故障能记录
- 版本能回滚
- 修改能验证

如果项目在这个阶段就追求完整架构、完整功能、完整优化、完整测试，往往会过早压垮自己。

### 2. 第二阶段：相持期

这是项目最漫长也最考验人的阶段。

典型现象：
- bring-up 结束后，问题并没有减少，反而开始系统性暴露
- 联调时，软件、硬件、协议、环境问题交织在一起
- 偶发 bug、时序 bug、并发 bug、边界 bug 轮番出现
- 新需求插入、旧缺陷返潮、版本回归问题并存
- 开发团队开始疲劳，管理层开始焦虑

这时候最容易出两种错：
- 要么悲观地认为项目没有希望
- 要么情绪化加班加人，试图“硬冲过去”

《论持久战》的启发是：
相持不是坏事，它往往意味着敌我态势正在变化。

翻译成工程语言，就是：
这个阶段真正重要的任务，是持续改变系统脆弱性分布。

具体包括：
- 提高系统可观测性
- 减少不可复现问题
- 建立回归测试基线
- 逐步固化接口
- 压低模块耦合度
- 提高故障定位效率
- 把“人肉经验”沉淀为团队能力

### 3. 第三阶段：反攻期

嵌入式工程的“反攻期”，不是简单意义上的结尾，而是项目从“能用”走向“可交付、可量产、可维护”的阶段。

典型任务包括：
- 性能优化
- 功耗优化
- 长时间运行稳定性提升
- 生产测试支持完善
- 现场故障恢复能力增强
- 版本冻结策略成熟
- 文档、手册、诊断工具链补齐

很多项目以为前两个阶段结束就算赢，其实这时只是“能打”，还不等于“能长期赢”。

真正的胜利，是产品具备了：
- 可复制
- 可验证
- 可量产
- 可诊断
- 可演进
- 可交接

## 四、“持久中的速决”是最适合工程人的一句话

《论持久战》中有一个非常适合工程项目的思想：
在战略持久中，争取战役和战斗的速决。

翻译成工程话，就是：
全局必须按长周期设计，但局部问题必须尽量短闭环解决。

这是长周期嵌入式项目最核心的节奏管理原则之一。

### 全局上为什么必须“持久”

因为你不能假设：
- 硬件一次无误
- 需求长期稳定
- 现场环境可完全模拟
- 驱动联调线性推进
- 量产后不会出现批量一致性问题

所以项目规划必须接受一个现实：
系统真正成熟，往往需要多轮迭代、长期消耗和分阶段收敛。

### 局部上为什么必须“速决”

因为如果每个具体问题都拖成长周期，项目会被细碎损耗拖死。

例如：
- 单个驱动故障，应尽快缩小到复现条件和根因假设
- 单个 crash，应尽量在短周期内形成最小复现与修复验证
- 单个通信异常，应快速明确是硬件、电气、时序还是协议栈问题
- 单个模块重构，应有明确边界、验收标准和退出条件

工程里最糟糕的状态，是：
- 全局节奏瞎快
- 局部问题瞎拖

正确做法恰恰相反：
- 全局上承认长期
- 局部上追求快打快收口

## 五、主动性、灵活性、计划性：嵌入式团队的核心素养

《论持久战》专门讲“主动性，灵活性，计划性”。
这三条几乎可以直接转化为嵌入式研发准则。

### 1. 主动性：不是被问题推着走

嵌入式工程里，主动性首先体现在“主动建设问题定位能力”。

比如：
- 主动加日志点，而不是出事后才补
- 主动设计 fault dump，而不是 crash 后只会重启
- 主动布置 trace，而不是靠猜测做时序分析
- 主动做看门狗、健康检查和恢复路径，而不是等死机现场教育人
- 主动整理知识和诊断手段，而不是把经验锁在个人脑子里

一个团队如果长期被问题追着跑，说明它失去了工程主动权。

### 2. 灵活性：不是摇摆，而是会换打法

嵌入式研发最怕“单一路线执念”。

比如：
- 明明真机调不动，还死守真机，不做模拟与 mock
- 明明板级问题明显，还坚持在应用层反复修补
- 明明只是临时止血问题，却硬要做全局重构
- 明明到了稳定期，还持续用实验性策略推动主版本

灵活性不是没有原则，而是能够根据阶段和问题性质切换策略。

真正成熟的团队会判断：
- 什么时候快速 patch
- 什么时候局部重构
- 什么时候隔离风险
- 什么时候推动板改
- 什么时候冻结变更
- 什么时候允许降级换取整体进度

### 3. 计划性：不是文档很多，而是关键路径清楚

嵌入式工程常见伪计划状态是：
- PPT 很完整
- 甘特图很漂亮
- 实际依赖关系却混乱
- 风险点没有前置识别
- 每个阶段退出标准不清

真正的计划性是：
- 依赖顺序清楚
- 关键路径清楚
- 里程碑和验收条件清楚
- 风险优先级清楚
- 板级与软件联调节奏清楚
- 哪些问题必须先解决，哪些可以延后清楚

一个好项目计划，不是为了汇报，而是为了让团队不在长周期中迷路。

## 六、“运动战、游击战、阵地战”对工程组织的启示

《论持久战》并不迷信单一作战形式，而强调不同作战方式的组合。这个思路对嵌入式研发组织很有参考性。

### 1. “运动战”对应主战线快速推进

适用于：
- bring-up
- 关键链路打通
- 主功能联调
- 跨模块主线问题集中突击

特点是：
- 快速集火
- 资源集中
- 为全局破局服务

### 2. “游击战”对应长尾问题治理和工具能力建设

适用于：
- 长尾 bug 清理
- 自动化脚本补齐
- 文档补齐
- 监控能力补齐
- 调试工具补齐
- 边缘模块问题持续压降

很多团队只重主线功能，不重“游击战”，结果项目后期四处漏风，交付看似完成，维护代价却极高。

### 3. “阵地战”对应稳定性固守和质量防线

适用于：
- 版本冻结
- 回归测试
- 关键接口保护
- 量产版本稳定维护
- 已知稳定模块防回退

如果项目从头到尾都只会“冲锋”，那后期大概率会死在回归和收敛阶段。

优秀的嵌入式团队，不是只会打一种仗，而是知道什么时候快攻，什么时候侧击，什么时候固守。

## 七、“消耗战”和“歼灭战”如何翻译成工程实践

这部分对编码工程特别有启发。

### 1. 消耗战：持续降低系统脆弱性

在工程里，很多工作都属于“消耗战”：
- 一点点降低 crash 率
- 一点点压低偶发异常比例
- 一点点提升自动化覆盖率
- 一点点清理日志盲区
- 一点点减少线上不可诊断问题
- 一点点清理历史遗留告警和不稳定接口

这些工作通常不耀眼，但它们决定项目是否能熬过长周期。

### 2. 歼灭战：对关键根因做彻底清除

并不是所有问题都值得打一场大仗，但对一些关键根因，必须发动“歼灭战”。

例如：
- 长期困扰系统的 DMA/cache 一致性问题
- 一个核心竞态导致的偶发死锁
- 一个架构级内存生命周期错误
- 一个长期无法定位的现场 crash 根因

如果这类问题一直靠补丁维持，系统将长期失血。

因此，高水平团队必须有能力判断：
哪些问题只需控制，哪些问题必须根除。

## 八、“兵民是胜利之本”对团队建设的提醒

如果把这句生硬直译进工程，当然不合适。
但其方法内核非常值得借鉴：
最终决定成败的，不只是少数高手，而是整体组织能力。

长周期嵌入式项目里，最危险的结构就是：
- 只有一个人会看启动链
- 只有一个人会改驱动
- 只有一个人掌握现场诊断方法
- 只有一个人知道量产脚本和测试工装

这类“单点英雄主义”在短期内效率很高，长期却极其脆弱。

真正稳的工程组织应做到：
- 关键模块知识有文档
- 关键调试手段可共享
- 关键故障定位路径可复制
- 至少两个人能接管关键组件
- 工具链与流程尽量制度化

长周期工程不是拼谁最能熬夜，而是拼团队整体是否具备持续作战能力。

## 九、对嵌入式编码工程师的具体建议

如果把本文压缩为可执行建议，我会给一线工程师以下十条：

1. 不要把“能跑”误认为“做完”
2. 先建可观测性，再谈复杂优化
3. 遇到问题先缩小边界，再扩大动作
4. 小问题短闭环，大问题分阶段
5. 优先处理高复发、高传播、高隐蔽性的缺陷
6. 核心链路宁可先简单，也不要过早花哨
7. 不要把单次联调成功当作系统稳定
8. 对关键根因要敢于彻底解决，不要永远贴补丁
9. 学会记录、复盘和沉淀，而不是只做“会修这一次”
10. 把自己从“单兵救火者”升级为“团队能力建设者”

## 十、也必须看到它对工程的边界

《论持久战》能给工程很强的方法启发，但不能机械照搬。

### 1. 工程不是战争

工程更强调：
- 证据
- 复现
- 验证
- 协作
- 稳定流程

而不是持续动员式的情绪高压。

### 2. 团队不能被“斗争化语言”绑架

如果把项目管理搞成天天“决战”“攻坚”“冲锋”，往往会带来：
- 信息失真
- 问题上报迟滞
- 人员透支
- 短期英雄主义压过长期工程建设

### 3. 工程最终仍要回到可验证性

不管叙事多么激昂，真正有价值的判断还是这些：
- 问题能否复现
- 修复能否验证
- 影响能否量化
- 风险能否隔离
- 经验能否复用

因此，最好的借鉴，是吸收其节奏感、阶段论和组织方法，而不是照搬其对抗性话语。

## 十一、结语

《论持久战》对于长周期嵌入式软件工程最深的启示，不是“咬牙坚持”四个字，而是更细致也更严厉的方法论要求：

- 要承认项目天然是长周期复杂系统
- 要接受阶段推进而非幻想一步到位
- 要在全局长期中争取局部速决
- 要用主动性、灵活性和计划性争取工程主动权
- 要把问题治理从个人英雄主义升级为团队化、制度化能力
- 要在长期消耗中识别值得发动“歼灭战”的关键根因

对嵌入式软件工程而言，真正的胜利从来不是某次 demo 漂亮、某次联调侥幸成功，而是系统最终变得可交付、可量产、可维护、可迭代，并且团队具备把这种能力持续复制下去的组织基础。

这，才是长周期工程真正意义上的“最后胜利”。
