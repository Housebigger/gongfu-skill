# 从《中共中央关于九月会议的通知》看嵌入式项目阶段复盘：用数据复盘、设定量化目标、为下个项目做好准备

日期：2026-06-12

原文对应：
《中共中央关于九月会议的通知》
原文链接：https://www.marxists.org/chinese/maozedong/marxist.org-chinese-mao-19481010.htm

写作背景：
1948年10月10日，毛泽东为中共中央起草了《中共中央关于九月会议的通知》，向全党通报1948年9月中央政治局会议（即"九月会议"）的主要决议。这篇通知不是一份泛泛的政治宣言，而是一份极其硬核的阶段复盘加下一阶段动员文件。毛泽东在通知中做了三件事：第一，用具体数据总结过去两年的战果——歼敌多少个旅、解放多少座城市、土地改革推进到什么程度、军队扩充到多少万人，全部用事实说话，没有一句空洞的"形势大好"。第二，根据这些事实，对当前阶段做出明确判断——解放战争已经到了"从根本上打倒国民党反动派"的转折点，但距离最终胜利还需要一定时间，不能因为形势好就松懈，也不能因为还要打就悲观。第三，给下一阶段设定了具体的量化目标和行动要求——再歼敌多少个师、再解放多少个地区、军队要怎么整编、财政经济要怎么统一、干部要怎么准备。整篇通知的结构就是：用数据复盘上一阶段 → 基于事实判断当前阶段 → 设定量化目标并部署下一阶段。这个结构，今天看起来仍然是一份标准的"阶段复盘 + 下一阶段OKR"模板。

说明：
把这篇文章转译到今天嵌入式项目的阶段复盘中，现实感极强。很多嵌入式团队在项目节点复盘、阶段总结或版本交付回顾时，最常见的问题不是不复盘，而是复盘没有数据、目标没有量化、下一阶段的准备没有落地的具体安排。复盘变成了"大家辛苦了"的鼓劲会，目标变成了"争取做得更好"的口号，下一阶段的准备变成了"我们接着干"的表态。131提醒嵌入式团队：真正有价值的阶段复盘，必须用数据说话——上个阶段交付了什么、测试通过率怎么样、核心缺陷趋势如何；基于这些数据判断当前阶段的位置——是离量产更近了还是更远了，决定项目成败的结构性问题是什么；然后为下一阶段设定明确的量化目标——下个阶段的缺陷密度要降到什么水平、关键模块要通过什么验证、团队要怎么调整。没有数据的复盘是空谈，没有量化目标的部署是空转。

核心思想的当代转译：
如果把《中共中央关于九月会议的通知》翻成今天嵌入式项目阶段复盘的语言，我会把它概括成一句话：项目阶段复盘必须用具体数据回顾上一阶段的交付结果，基于事实判断当前进度和结构性问题，然后为下一阶段设定明确的量化目标和资源部署——不用数据的复盘是空谈，不设量化目标的部署是空转。

## 一、第一层提醒：复盘的第一步不是谈感受，而是把数据摆出来

毛泽东在通知里做的第一件事，不是分析形势、不是鼓舞士气，而是摆数据。歼敌数字、解放城市数量、军队规模、土改进度、财政状况——全部用具体数字说话。

翻到嵌入式项目的阶段复盘里，这句话极其值钱。因为绝大多数复盘会的开场是这样的：

- "这个阶段大家都很拼，加班很多，辛苦了。"
- "整体来说，板子跑起来了，基本功能没问题。"
- "客户那边还算满意，没有出什么大事。"
- "大部分模块都交付了，还有几个延到了下个阶段。"

这些话的共同特点是什么？没有数据。

真正有价值的复盘开场应该是这样的：

- 这个阶段计划交付5个模块，实际完成4个，电源管理模块因BOM变更延期两周，延期原因是……
- 系统集成测试用例执行了326条，通过289条，通过率88.7%，未通过的37条中，崩溃类11条、性能不达标15条、功能偏差11条；
- 客户现场反馈问题18个，已关闭12个，遗留6个，其中P1级别2个，集中在通讯模块的偶发丢包；
- 固件版本迭代了7次，平均每个版本的回归测试周期从5天缩短到3天，但自动化覆盖率只从42%升到45%；
- BOM成本从目标价85元优化到82.3元，但关键物料交期从8周拉长到12周，供应链风险上升。

131提醒嵌入式团队：
复盘的第一步不是谈感受，而是把数据摆出来。
没有数据的复盘，本质上是在用印象代替事实。
"感觉还行"和"数据证明还行"，是完全不同的两件事。
数据不需要完美，但必须要有。
粗略的数据也比精准的印象可靠。

## 二、第二层提醒：数据不是用来粉饰太平的，而是用来暴露结构性问题的

毛泽东在通知里不只报喜，也报了实际问题：解放区财政困难、干部数量不足、某些地区的土改执行有偏差。这些不是用来"表态承认困难"的装饰，而是用来识别下一阶段要解决的结构性矛盾。

翻到嵌入式项目的复盘中，这层意思对应的是一个极其普遍的问题：很多团队的数据复盘，变成了"挑好看的数据讲"：

- 功能演示通过了，大讲特讲；压力测试下系统崩溃了，一笔带过；
- 模块级测试覆盖率达到90%，大讲特讲；系统集成测试覆盖率只有35%，不在复盘会上讨论；
- 客户验收签字了，大讲特讲；产线上不良率偏高的问题，归因到"产线操作不规范"就过去了；
- 单板功耗达标了，大讲特讲；高温环境下死机偶发的问题，标注为"待复现"就搁置了。

这种"选择性呈现数据"的复盘，和不做复盘差不多——因为真正决定项目下一阶段走向的，往往不是那些好看的数据，而是那些被跳过的问题。

131提醒嵌入式团队：
数据的价值不在于证明"我们做得不错"，
而在于暴露"我们在哪里还有结构性问题"。
功能通了但压力测试崩溃，说明系统的健壮性不够；
模块测试过了但集成测试大面积失败，说明模块间的接口定义有问题；
客户验收了但产线不良率高，说明工程化设计没有充分考虑可制造性。
复盘数据的时候，最值得关注的数据，
恰恰是那些"不好看"的数据。
因为那些数据背后，藏着下一阶段最需要解决的问题。

## 三、第三层提醒：基于数据判断当前阶段位置——是离量产更近了，还是只是更忙了

毛泽东在通知里做了一件非常关键的事：基于战果数据，判断当前解放战争处于什么阶段。他的结论是：已经到了从根本上打倒国民党的转折点，但还没有到最终胜利，接下来还要继续打硬仗。

这个判断的厉害之处在于：它既不是盲目乐观（"我们快赢了"），也不是保守悲观（"还要打好久"），而是基于事实的阶段定位。

翻到嵌入式项目的复盘中，这层意思解决的是一个常见问题：团队分不清"更忙了"和"更接近量产了"。

很多团队复盘时的表述是：

- "我们这个阶段做了很多调试。"——做了很多调试，但离量产的标准还差多远？
- "团队加班很多，很辛苦。"——辛苦不等于有进展，忙不等于在靠近目标；
- "所有模块都跑通了。"——跑通了，但可靠性测试、环境测试、量产验证做了没有？
- "客户现场比上个阶段稳定了。"——稳定了多少？核心缺陷解决了没有？偶发问题定位了没有？

131提醒嵌入式团队：
阶段复盘的核心判断，不是"我们做了多少事"，
而是"我们离量产更近了多少"。
调了20个bug但系统级的偶发问题还在，说明这20个bug可能不是决定性的；
加班很多但关键路径上的问题没解决，说明忙在了非关键方向上；
模块都跑通了但量产测试还没开始，说明离真正的量产还有一段路。
真正有价值的阶段判断只有一个标准：
这个阶段结束后，项目距离量产是不是真的更近了。
如果答案是"是的"，数据会证明；
如果答案是"不确定"，那说明上一阶段的目标和衡量标准本身就有问题。

## 四、第四层提醒：下一阶段必须设定量化目标，不能只有方向没有刻度

毛泽东在通知里最硬的部分，是他给下一阶段设定了极其具体的量化目标：再歼敌多少个旅、再解放多少个省、军队要扩充到多少人、干部要准备多少名。每一个目标都有数字，不是"继续努力"，不是"加大力度"。

翻到嵌入式项目的复盘中，这对应的是几乎所有团队都犯过的错：复盘完了，下一阶段的目标写成：

- "提升系统稳定性。"——稳定性到什么程度？MTBF达到多少小时？crash率降到多少？
- "优化功耗。"——优化到什么数值？待机电流降到多少毫安？满载功耗控制在多少瓦？
- "推进量产进度。"——推进到哪个节点？产测通过率达到多少？直通率目标是什么？
- "加强质量管控。"——怎么衡量？缺陷密度降到多少？回归测试覆盖率提到多少？

这些没有数字的目标，和没有目标是一样的。因为没有人知道"做到了"长什么样，也没有人知道"没做到"该追谁的责。

131提醒嵌入式团队：
下一阶段的目标，必须是量化的。
"提升系统稳定性"不是目标，"系统连续运行MTBF从200小时提升到500小时"才是；
"优化功耗"不是目标，"待机电流从15mA降到5mA以内"才是；
"推进量产"不是目标，"产测直通率从85%提升到95%"才是。
量化目标的本质，不是给团队加压力，
而是让所有人知道"做成了"的标准是什么。
没有刻度的方向，不是方向，是口号。
有刻度的方向，才能变成团队共同的行动标尺。

## 五、第五层提醒：量化目标不只是数字，更要拆到具体的行动部署和责任人

毛泽东在通知里不只是说"再歼敌一百个旅"就完了，他还部署了具体的行动：各个野战军分别负责哪个方向、军队怎么整编、财政经济怎么统一、干部怎么培训和分配、新区工作怎么开展。每一个量化目标后面，都跟着具体的行动安排和责任人。

翻到嵌入式项目，这层意思对应的是：很多团队定了量化目标之后，就以为工作做完了。目标写在项目计划里、贴在war room墙上、发在项目群里，但没有拆解成具体的行动和责任人。

现实中的场景：

- 目标是"产测直通率提升到95%"，但没有指定哪些测试项是不通过的重灾区、哪个模块要重新设计、谁负责优化测试方案；
- 目标是"系统MTBF提升到500小时"，但没有拆到具体是哪个模块的偶发问题要解决、哪些边界条件要加固、谁负责长期可靠性测试；
- 目标是"待机电流降到5mA"，但没有说哪个子系统是功耗大户、哪个外设的休眠策略要重做、谁负责功耗分析和优化；
- 更常见的是：量化目标定完了，但资源分配没有变化——人还是那些人、设备还是那些设备、排期还是那个排期，目标变成了挂在墙上的数字。

131提醒嵌入式团队：
量化目标只是起点，不是终点。
目标定了之后，必须回答三个问题：
第一，这个目标要通过哪些具体行动来实现？
第二，每个行动由谁负责、什么时间完成？
第三，现有的资源分配需不需要调整来支撑这些行动？
如果目标定了但行动没变、资源没动、人不调整，
那这个目标就不是用来执行的，而是用来汇报的。
真正有效的阶段部署，是目标 + 行动 + 责任人 + 资源调整四位一体。

## 六、第六层提醒：阶段复盘不只是向后看，更是为下一阶段做好准备——团队状态、能力储备和工具链升级

毛泽东在通知里有一大段内容不是谈军事战果，而是谈组织准备：干部要准备多少、怎么训练、怎么分配；军队要怎么整编、怎么提升正规化水平；新解放区的接收和管理要怎么准备。他非常清楚：下一阶段的仗能不能打赢，不只取决于上一阶段的战果，还取决于组织有没有为下一阶段做好准备。

翻到嵌入式项目，这层意思是最容易被忽视的：

大多数团队的阶段复盘，只关注"上个阶段做了什么"和"下个阶段做什么"。但很少有人问：

- 团队的整体状态怎么样？连续高强度调试和驻场之后，团队是精力充沛还是已经透支？下一阶段需要驻场的时间是不是更长了？
- 团队的能力储备够不够？下一阶段如果有新的认证要求（比如EMC整改）、新的供应商平台、新的制造工艺，现有的技能能不能支撑？
- 调试设备和工具链需不需要升级？上一阶段暴露出的效率瓶颈——示波器不够用、产测夹具不完善、固件烧录工具不稳定——下一阶段能不能解决？
- 知识和经验有没有沉淀？上一个阶段踩过的坑、定位过的疑难问题、积累的调试经验，有没有被记录到工程文档里，还是只存在几个工程师的脑子里？
- 供应链关系需不需要调整？上一阶段的物料延迟、替代料验证、供应商配合度问题，下一阶段有没有预案？

131提醒嵌入式团队：
阶段复盘的价值，不只是总结过去，
更是为下一阶段做好准备。
产品要准备，团队也要准备。
技术要准备，工具链也要准备。
上一阶段的胜利，不自动带来下一阶段的胜利。
如果团队状态透支了、能力没跟上、工具链没升级、供应链没预案，
那下一阶段开始的时候，上一阶段积累的优势就会被消耗掉。
真正成熟的团队，在复盘的时候会花同样的精力
回答两个问题：上一阶段做对了什么？下一阶段我们准备好了吗？

## 结语：131给嵌入式工程的硬提醒，是用数据复盘、设定量化目标、为下一阶段做好全面准备

所以，131对今天嵌入式团队最现实的提醒是：项目阶段复盘不能只谈感受，必须用数据说话。数据不是为了粉饰太平，而是为了暴露结构性问题。基于数据判断当前阶段的位置——是离量产更近了，还是只是更忙了。下一阶段必须设定量化目标，不能只有方向没有刻度。量化目标要拆到具体行动、责任人和资源调整，不能只挂在墙上。阶段复盘不只是向后看，更是为下一阶段做好全面准备——团队状态、能力储备、工具链升级、供应链预案，每一项都要检查。不用数据的复盘是空谈，不设量化目标的部署是空转，不做准备就开始的下一阶段是用上一阶段的积累为下一阶段的失误买单。真正有效的阶段复盘，是用事实校验信心，用数据识别问题，用量化目标锁定方向，用行动部署和资源调整确保执行，最后用团队、工具和供应链的全面准备为下一阶段的胜利打底。
