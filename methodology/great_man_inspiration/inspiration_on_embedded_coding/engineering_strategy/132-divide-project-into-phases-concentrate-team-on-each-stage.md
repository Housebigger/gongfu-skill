# 从《关于淮海战役的作战方针》看项目攻坚：把大项目拆成阶段，每阶段集中团队攻克一个核心技术难点，用主要精力屏蔽外部干扰

写作背景：
1948年10月11日，毛泽东为中央军委起草了《关于淮海战役的作战方针》，这是一份极其精密的作战计划书。这份方针的核心不是一句"打就完了"的豪言壮语，而是一套把大战役拆成三个阶段、每个阶段集中力量打一个硬仗的完整打法。毛泽东做了几件极其漂亮的事：第一，把整个战役拆成三个阶段，每个阶段有明确的中心目标——第一阶段集中力量打黄百韬兵团，第二阶段打黄维兵团，第三阶段打杜聿明集团，三个阶段依次推进、环环相扣。第二，明确规定了兵力使用的核心原则——集中两倍于敌的优势兵力歼灭当面之敌，同时用一半以上的总兵力去阻击和牵制敌方增援部队。第三，对每个阶段的时间做出了预估——每个阶段大约2-3周，总计一个半月到两个月。第四，强调要借鉴之前济南战役、豫东战役中已经验证过的打法，不搞花哨创新，用经过实战检验的战术。第五，在制定当前作战方案的同时，就已经规划好了从淮海战役到后续渡江战役、乃至解放全中国的完整路线图。更值得深思的是，这份方针的预判极其精准——实际的淮海战役进程基本遵循了这份计划，而且战果远超预期。把一份战役方针翻成今天嵌入式大项目的管理语言：不要试图一口气把整个项目做完，而是把项目拆成几个阶段，每个阶段集中团队最核心的力量攻克一个技术难点，同时用主要精力去屏蔽外部干扰和需求变更，用已经验证过的方案降低风险，并且从一开始就要有从当前阶段一直到量产交付的完整路线图。

说明：
把这篇文章转译到今天嵌入式项目的阶段攻坚中，现实感极强。很多嵌入式团队在面对一个大项目时，最常见的错误不是不努力，而是不分行——整个团队同时铺在所有模块上，没有主攻方向，没有阶段划分，所有人都在忙但没有人在关键路径上形成突破。结果就是项目做了三个月，好像每个模块都在推进，但没有一个模块真正收敛。需求变更从四面八方涌进来，团队疲于应付，核心技术难题反而没有人集中攻坚。132提醒嵌入式团队：真正高效的项目推进方式，是把大项目拆成几个阶段，每个阶段集中最核心的人力攻克一个技术难点，同时用足够的力量去阻挡外部需求变更和干扰。不分阶段的攻坚是蛮干，不集中力量的推进是散打，不屏蔽干扰的项目是被需求牵着鼻子走。

核心思想的当代转译：
如果把《关于淮海战役的作战方针》翻成今天嵌入式项目攻坚的语言，我会把它概括成一句话：大项目必须拆成阶段来打，每个阶段集中核心团队攻克一个技术难点，用足够的力量屏蔽外部干扰和需求变更，用已验证的方案降低风险，并且从一开始就要规划好从当前阶段一直到量产交付的完整路线图——不分阶段的攻坚是蛮干，不集中力量的推进是散打。

## 一、第一层提醒：大项目不能一口气吞下去，必须拆成阶段，每个阶段有明确的中心目标

毛泽东在方针里做的第一件事，不是分析敌我形势，不是鼓舞士气，而是把整个淮海战役拆成三个阶段：第一阶段打黄百韬，第二阶段打黄维，第三阶段打杜聿明。三个阶段各有一个中心目标，依次推进，不打乱仗。

翻到嵌入式大项目的管理里，这句话极其值钱。因为绝大多数团队在面对一个大项目时的做法是：

- "这个项目有8个模块，我们8个模块同时开工。"
- "硬件、固件、上位机、测试并行推进，齐头并进。"
- "所有功能同时开发，争取一次性交付。"
- "先把所有模块都做到60%，然后一起集成优化。"

这些做法的共同问题是什么？没有主攻方向。团队的力量被摊薄在所有方向上，每个方向都在做，但没有一个方向做到位。

真正高效的项目推进应该是这样的：

- 第一阶段（第1-4周）：集中80%的固件力量攻克底层驱动和RTOS移植，确保芯片平台跑通、外设驱动稳定，这是后面所有功能的地基；
- 第二阶段（第5-8周）：核心团队转向通讯协议栈和核心算法实现，同时用部分人力维护已跑通的平台层、处理遗留问题；
- 第三阶段（第9-12周）：集中力量做系统集成和可靠性测试，把前两个阶段的成果拧成整体，同时处理客户侧的需求对齐和认证准备。

132提醒嵌入式团队：
大项目不能一口气吞下去，必须拆成阶段来打。
每个阶段有一个明确的中心目标——这个阶段最核心的技术难点是什么，团队最集中力量要攻克的是什么。
没有中心目标的阶段，不叫阶段，叫"这段时间大家都在忙"。
没有阶段划分的项目，不叫项目管理，叫"一拥而上"。
八个模块同时开工，看起来热火朝天，实际上哪个模块遇到硬骨头的时候，都没有足够的人力去啃下来。
先集中力量啃下一块硬骨头，再啃下一块，最后把所有成果串起来——这才是高效的项目推进方式。

## 二、第二层提醒：集中优势力量攻坚——每打一个硬仗，要用两倍于问题的资源去解决它

毛泽东在方针里规定了一个硬原则：歼灭当面之敌时，必须集中两倍于敌的兵力——一个敌军师，用两个纵队去打；一个敌军旅，用两个师去打。不是一对一地拼消耗，而是用绝对优势的力量形成碾压。

翻到嵌入式项目里，这句话对应的现实极其刺痛。因为绝大多数团队在面对核心技术难题时，资源分配是什么样的？

- 功耗优化是关键难题，但只安排了一个工程师"兼着做"；
- EMC整改过不了认证，但团队里只有一个人懂射频，其他人在等；
- 无线协议栈稳定性有问题，但负责这个模块的工程师同时在改另外三个模块的bug；
- 系统偶发死机的问题定位了三周还没结果，但负责调试的工程师每周还要参加四个和这个问题无关的会议。

这些场景的共同本质是：面对硬骨头的时候，没有集中足够的力量。

真正有效的攻坚资源配置应该是：

- 功耗优化被定为当前阶段的核心目标，那么至少安排两个工程师全职投入——一个人做测量和分析，一个人做方案设计和验证，两周之内必须出结果；
- EMC整改过不了认证，那么把硬件团队里最懂信号完整性和射频的两个人集中在这个问题上，其他工作暂时移交，直到整改通过；
- 协议栈稳定性问题，把通讯模块相关的所有开发集中到一个攻坚小组，把非核心的需求变更暂时挡在外面，直到问题定位并解决。

132提醒嵌入式团队：
打硬仗的核心原则不是"咬牙坚持"，而是"集中碾压"。
一个核心技术难题，用两倍的人力和时间资源去攻克，不是浪费，而是效率。
因为一个硬骨头啃一周啃不下来，啃两周啃下来了，比一个人啃了六周还没啃下来要高效得多。
集中力量的本质，不是加人，而是让解决问题的人不再被其他事情分散精力。
一个工程师全职攻坚一个难题，比三个工程师各自兼着一小块要有效十倍。

## 三、第三层提醒：用一半以上的力量屏蔽外部干扰和需求变更——阻击战比攻坚战更花人力

毛泽东在方针里有一个极其容易被忽视但极其关键的部署：在集中力量歼灭当面之敌的同时，要用一半以上的总兵力去阻击和牵制敌方增援部队。也就是说，真正花在"打主攻"上的力量，可能不到总兵力的一半；另一半甚至更多，是用来"挡住别人不让别人干扰主攻"的。

翻到嵌入式项目管理中，这层意思是最被低估的。因为绝大多数项目经理的资源分配逻辑是：把所有人力都分配到开发任务上，没有人专门负责"挡住干扰"。

现实中的场景：

- 阶段核心目标是攻克协议栈稳定性，但客户每周都在提新需求，开发团队花30%的时间在处理需求评估和影响分析上；
- 核心攻坚正在关键时刻，但公司内部的质量审核、代码评审、过程文档占用了核心工程师大量时间；
- 团队在集中解决功耗问题，但销售带回来一个"紧急定制需求"，两个核心工程师被抽走去评估可行性；
- 攻坚小组正在调试偶发死机问题，但每周的跨部门协调会、项目汇报会、供应商对接会照常进行，没有人挡。

这些干扰看起来每一件都是"不得不做的事"，但加在一起，核心攻坚的力量就被稀释了。

真正有效的项目推进，应该这样做：

- 项目经理或技术负责人的核心职责之一，就是"挡住干扰"——不是所有需求都要接，不是所有会议都要参加，不是所有变更都要当天响应；
- 每个攻坚阶段，至少安排一个资深工程师或项目经理专门负责需求过滤、变更管理、对外沟通——他的任务不是写代码，而是确保写代码的人不被打扰；
- 对客户说"这个阶段我们在解决A问题，B需求记录下来，下个阶段处理"——而不是每个需求都"好的我们看看"；
- 对公司内部说"攻坚期间，核心人员不参加非必要会议，进度汇报由项目经理统一做"——而不是每个工程师都去开会汇报自己做了什么。

132提醒嵌入式团队：
攻坚战花的是人力，阻击战花的也是人力。
而且阻击战花的人力，往往比攻坚战更多。
一个项目经理如果能挡住80%的外部干扰，让核心团队专注攻坚，
他的贡献比写一万行代码还大。
很多项目推不动，不是因为技术难题解不了，
而是因为解技术难题的人没有时间解技术难题。
把一半以上的项目管理精力用来屏蔽干扰，不是浪费，是最高效的资源分配。

## 四、第四层提醒：用已经验证过的方案降低风险——不搞花哨创新，用实战检验过的打法

毛泽东在方针里特别强调：打法要借鉴济南战役、豫东战役中已经验证过的经验——怎么围点打援、怎么穿插分割、怎么集中优势兵力。不是每次都发明新战术，而是把已经被实战证明有效的打法直接拿来用。

翻到嵌入式项目里，这层意思对应的是很多团队爱犯的一个毛病：遇到问题的时候，不先用已经验证过的方案，而是喜欢尝试新技术、新方案、新架构。

现实中的场景：

- 之前的量产项目用的是FreeRTOS，稳定可靠，这个项目突然换成Zephyr，理由是"新技术更有前途"——然后花了两个月在移植和踩坑上；
- 之前的电源管理方案用的是成熟的DC-DC + LDO架构，验证过EMC和热设计，这个项目突然换成全新的PMIC方案，理由是"集成度更高"——然后花了三周在PMIC的配置和稳定性问题上；
- 之前的通讯协议用的是团队自研的轻量级协议，已经在三个项目上稳定运行，这个项目突然换成某大厂的开源协议栈，理由是"功能更丰富"——然后花了六周在协议栈的适配和稳定性调试上；
- 之前的测试框架用的是团队熟悉的Python + pyserial，这个项目突然换成某商业测试平台，理由是"界面更专业"——然后测试工程师花了一个月学习新平台，测试效率反而下降。

这些决策的共同特点是：对"新方案"的收益有想象，对"旧方案"的可靠性缺乏尊重。

132提醒嵌入式团队：
每个阶段的核心目标已经足够有挑战性了，
不要在方案选择上再给自己增加不确定性。
能用验证过的方案，就不要换新方案。
能用跑通过的架构，就不要重新设计。
能用团队熟悉的工具链，就不要引入新的工具。
项目攻坚的阶段，最不需要的就是"额外的不确定性"。
创新的精力应该花在核心技术难题上，而不是花在替换已经被证明有效的方案上。
把实战检验过的打法直接拿来用，不是保守，是效率。
把精力留给真正需要创新的地方，才是对团队时间的尊重。

## 五、第五层提醒：每个阶段2-3周，总时长心中有数——时间预判让团队知道自己在路上哪个位置

毛泽东在方针里对时间有非常清晰的预判：每个阶段大约十几天到二十天，总计一个半月到两个月。这个时间预估不是拍脑袋，而是基于之前的战斗经验做出的判断。

翻到嵌入式项目里，这层意思对应的是一个极其普遍的问题：很多团队的项目计划，要么只有最终交付日期，没有阶段节点；要么阶段节点的时间跨度太大，失去了节点控制的意义。

现实中的场景：

- 项目计划只写了"三个月后量产"，中间没有阶段节点，到了第二个月才发现进度严重滞后；
- 阶段节点的时间跨度是"第一阶段两个月，第二阶段两个月"，太长了，团队在第一阶段前六周感觉时间还多，第七周才开始着急；
- 每个阶段的时间预估是凭直觉定的，没有参考之前类似项目的实际耗时，结果每个阶段都比计划超出50%以上；
- 更常见的是：阶段划分是做了，但没有告诉团队"我们预计这个阶段多长"，导致团队不知道自己是在正常推进还是已经落后了。

132提醒嵌入式团队：
每个阶段的时间预估，不是用来束缚团队的枷锁，
而是让团队知道自己"在路上哪个位置"的里程碑。
2-3周一个阶段，是一个合理的节奏：
足够长以完成一个有意义的攻坚目标，
足够短以保持团队的紧迫感和节奏感。
如果一个阶段超过一个月还没有阶段性成果，
要么是阶段划分太粗了，要么是遇到了比预期更大的难题——
不管是哪种，都需要及时调整，而不是闷头继续。
时间预估不一定要准，但一定要有。
有预估，团队才知道自己是提前了还是落后了。
没有预估的计划，不是计划，是许愿。

## 六、第六层提醒：从当前战役到最终胜利的完整路线图——不只看这一仗，更要看到整场战争

毛泽东在方针里做了一件超越绝大多数军事计划的事：在制定淮海战役的作战方案时，他已经规划好了从淮海战役到渡江战役、再到解放全中国的完整路线图。淮海战役不是一场孤立的仗，而是整个解放战争的承前启后之战。

翻到嵌入式项目里，这层意思解决的是一个战略层面的问题：很多团队只看当前阶段，没有完整的路线图。

现实中的场景：

- 团队在做原型验证，但没有想过原型验证之后的小批量试产要怎么做、需要什么资源、时间节点在哪里；
- 阶段攻坚热火朝天，但没有规划好攻坚结束后怎么过渡到量产——测试方案、产线设备、生产夹具、SOP文档，全部还没开始准备；
- 只看到当前的认证整改，没有规划好认证之后的市场导入、客户培训、售后支持怎么做；
- 更常见的是：项目经理的所有精力都在解决当前阶段的问题，没有人抬头看路——当前阶段做完了，下一个阶段怎么接？接不上的话，当前阶段的成果就在等待中折损。

132提醒嵌入式团队：
真正优秀的项目规划，不只看当前这一仗，
更要看到从当前阶段到最终量产交付的完整路线图。
当前阶段攻克的是协议栈稳定性，但路线图上要能看到：
协议栈稳定之后是系统集成测试，系统集成之后是小批量试产，
试产之后是量产爬坡，爬坡之后是稳定交付。
每个阶段之间怎么衔接、需要什么资源、可能有什么风险，
不需要现在就全部想清楚，但必须现在就有一个框架。
有了路线图，团队才知道当前的攻坚在整个项目中的位置。
没有路线图的攻坚，是打一仗看一仗。
有路线图的攻坚，是打一仗进一步。

## 结语：132给嵌入式工程的硬提醒，是把大项目拆成阶段、集中力量、屏蔽干扰、用验证过的方案、从当前看到最终

所以，132对今天嵌入式团队最现实的提醒是：大项目必须拆成阶段来打，每个阶段有一个明确的中心目标。面对核心技术难题，要集中两倍于问题难度的资源去碾压，不要一对一地拼消耗。用足够的力量去屏蔽外部干扰和需求变更——阻击战比攻坚战更花人力。方案选择上，优先用已经验证过的打法，不搞花哨创新，把不确定性降到最低。每个阶段2-3周，总时长心中有数，让团队知道自己路上在哪个位置。从当前阶段到最终量产交付，要有完整的路线图，不打一仗看一仗。不分阶段的攻坚是蛮干，不集中力量的推进是散打，不屏蔽干扰的项目是被需求牵着鼻子走，没有路线图的计划是走一步看一步。真正高效的项目推进，是把大目标拆成阶段、把团队力量集中到核心、把干扰挡在门外、用验证过的方案降低风险、用时间预判控制节奏、用完整路线图指引方向——从第一仗打到最后一仗，仗仗有中心，步步有目标。
