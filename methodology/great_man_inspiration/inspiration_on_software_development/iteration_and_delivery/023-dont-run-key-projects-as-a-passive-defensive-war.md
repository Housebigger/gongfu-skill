# 从《和英国记者贝特兰的谈话》看软件团队别把关键项目打成“只有管理层和少数骨干硬扛”的被动防守战：要把更多角色组织进来，并从补洞转向主动进攻

日期：2026-04-16

原文对应：
《和英国记者贝特兰的谈话》

写作背景：
1937年10月25日，毛泽东在延安同英国记者贝特兰谈话。这篇谈话一方面回顾了抗日战争爆发以来的形势与教训，另一方面明确指出：战争虽然已经开始，但它仍然存在一个重大弱点——“抗战还只是政府和军队的抗战，不是人民的抗战”。因此，真正的胜利不能只靠少数上层和少数主力部队，而要通过政治制度改革、群众动员、统一指挥、战略战术调整，把战争变成真正更广泛参与的战争。与此同时，毛泽东也强调必须从“单纯防御”转向“积极攻击敌人的方针”，并特别总结八路军依靠独立自主的游击战和运动战、依靠“官兵一致、军民一致、宽待俘虏”的政治工作来形成更强战斗力。

说明：
今天把这篇谈话转译到软件团队语境里，当然不能照搬战争语言。现代软件组织面对的是关键项目、系统升级、市场窗口、交付压力、跨团队协同和资源有限，不是军事冲突。但这篇文章对软件团队特别有启发：
很多组织以为只要管理层拍板了、几个 TL 和核心工程师顶上了，项目就会自然推进。可现实里，很多关键项目之所以越做越累，不是因为方向完全错，而是因为它依然只是“管理层和少数骨干的抗战”，还不是整个产品工程系统真正参与的战役；同时，团队又长期停在被动补洞、被动防守、被动回应问题的状态，没有真正转向主动进攻。

核心思想的当代转译：
如果把这篇文章翻成今天软件团队的语言，我会把它概括成一句话：
关键项目想打赢，不能只靠少数 leader 和骨干工程师把它顶成一场长期防守战；你必须把更多真正相关的角色组织进来，建立更适配新阶段的治理和指挥方式，并从“持续补洞”升级到“主动攻击决定性瓶颈”。

## 一、第一层提醒：很多关键项目之所以久攻不下，是因为它还只是“管理层和骨干的战争”

这篇谈话里最有穿透力的一句判断之一，是“抗战还只是政府和军队的抗战，不是人民的抗战”。

翻到软件团队里，我觉得几乎可以直接变成：
很多关键项目，还只是管理层、项目负责人和少数核心工程师的战争，不是整个产品工程系统真正参与的战争。

这在现实里特别常见：
- 高层知道项目重要，但更多执行角色并不知道为什么重要
- 产品和研发负责人很紧张，但测试、设计、数据、客服、实施、运维并没有真正进入同一战役
- 核心骨干天天救火，外围团队却还按旧优先级和旧 KPI 运转
- 管理层一再强调战略级，实际支持系统却并没有形成

于是组织会出现一种非常熟悉的危险状态：
少数人极其焦虑，
多数人并没有真正进入。

这种项目当然也能推进一点，
但会非常脆。
因为所有压力都集中打在少数人身上，任何新的依赖、任何新的事故、任何一次资源抽调，都会让局面迅速恶化。

## 二、第二层提醒：真正的“项目群众性”，不是大家都知道有这件事，而是更多角色知道自己为什么进来、怎么进来

毛泽东为什么强调“唤起民众”？
因为他非常清楚：
大仗不能靠少数人闭门作战，必须让更广的力量看见自己进入的位置与意义。

放到软件团队里，“群众性”并不是人人来开会，也不是群里所有人都知道项目代号。
它更接近于：
- 设计知道为什么这次体验链路必须升级
- 测试知道为什么这次质量门槛必须抬高
- SRE 知道为什么这次稳定性不是附属项，而是主任务
- 客服和实施知道自己带回的一线问题会被当成关键输入
- 销售和运营知道哪些旧承诺必须让位给主战场
- 依赖方知道自己的接口动作为什么必须改变

也就是说：
真正的广泛参与，不是通知覆盖更广，
而是让更多相关角色明确：
这场仗和我有什么关系，我应该以什么方式进入，我的动作如何影响全局。

如果这一步做不到，所谓“公司级项目”通常只是少数人的幻觉，不是组织级能力。

## 三、第三层提醒：阶段已经变了，旧流程、旧权限、旧协作方式不改，新项目就只能被老系统拖死

这篇谈话很锋利的一层，是指出局势已经变化，但旧制度还没变化，于是“没有变化的东西和变化了的东西不相适应”。

这句话用来照今天的软件组织，非常准。

很多团队明明已经进入关键项目阶段，却还用普通迭代期的机制打仗：
- 项目已经是主战场，审批链却还是原来的速度
- 风险已经是系统级，升级路径却还是靠私聊和关系推动
- 依赖已经跨部门，资源分配却仍然平均主义
- 项目已经需要一线反馈直达，信息通道却仍然层层过滤
- 目标已经从“做功能”变成“打赢关键战”，考核和协作却还停在各自本位

于是你会看到一种非常别扭的状态：
人人都承认项目重要，
但组织的真实动作却还是旧世界。

这也是为什么很多项目明明有强共识，却依然进展缓慢。
不是没人懂，
而是旧系统没退场。

## 四、第四层提醒：很多团队的真正问题，不是还不够努力，而是一直在打“单纯防御”

毛泽东在谈话里明确说，很多军事上的失利，一个原因在于打的大半都是“单纯防御”的仗；而要胜利，就必须把方针改成“积极攻击敌人的方针”。

这对软件团队特别重要。

很多关键项目的日常状态，其实正是“单纯防御”：
- bug 来了就修 bug
- 报警来了就灭报警
- 客诉来了就补说明
- 依赖炸了就临时协调
- 数据跌了就写复盘
- 项目卡住了就加会、催人、补流程

这些动作是不是完全没价值？
当然不是。
但如果团队长期停在这里，项目就只是在“别出大事”的逻辑里打转。

真正能扭转局面的，往往不是防守动作本身，
而是主动进攻决定性瓶颈：
- 主动拆掉最关键的技术堵点
- 主动重排依赖关系
- 主动把一线最痛的链路打穿
- 主动收缩次要需求，为主任务形成兵力优势
- 主动设计新的交付节奏和治理路径

很多项目并不是死在问题太多，
而是死在团队太久没有从“被动应对问题”升级到“主动塑造局面”。

## 五、第五层提醒：关键项目需要一种“民主集中制”式治理——让真实信号能上来，让统一决断能落下去

谈话里有一段非常重要，毛泽东解释“民主集中制”：
它既要民主，又要集中；既要让真实力量表达出来，又要形成能够统一行动的政府与制度。

这对软件团队几乎就是关键项目治理的核心。

很多项目失败，常常卡在两种极端之间：

一种是只有集中没有民主：
- 上面直接拍板
- 下面只负责执行
- 风险和真实约束很难上来
- 一线知道哪里不行，但进不了判断层

另一种是只有民主没有集中：
- 谁都能提意见
- 会越开越多
- 信息越来越全，决策越来越慢
- 责任越分越散，最后没人真正拍板

真正适合关键项目的治理，应该是：
- 一线风险能快速上行
- 用户与系统真实信号能被及时吸收
- 关键依赖能被明确暴露
- 但在该收束的时候，有统一指挥和统一节奏

这其实就是软件项目里的“民主集中制”：
不是让组织变成混乱的意见市场，
也不是让项目沦为少数人闭门命令的黑箱，
而是先把真实情况接进来，再把决断收束成统一行动。

## 六、第六层提醒：不要再把所有工作都做成笨重的正面阵地战，要保留“游击战和运动战”能力

毛泽东在谈话里谈八路军战法时，强调“独立自主的游击战和运动战”，强调避免“一切被动的呆板的战法”。

我觉得这对软件团队非常值钱。

现实里很多团队一遇到大战，就会本能地把所有动作都做成重型阵地战：
- 什么都想一次性规划全
- 什么都要统一到全量方案后再动
- 什么问题都想通过更大更重的流程压住
- 一切动作都追求“从总部正面推进”

结果往往是：
组织越来越重，
节奏越来越慢，
真正可打穿问题的灵活性越来越差。

而“游击战和运动战”在软件团队里，常常意味着：
- 允许围绕某个高价值瓶颈快速成立小分队
- 允许先在一个局部链路打穿再扩面
- 允许用更轻的原型或更机动的验证方式先拿下侧翼突破口
- 允许绕开最笨重的正面阻力，先改变敌我条件

这不是鼓吹混乱，
而是提醒团队：
关键阶段不能只会正面硬顶，必须保留纪律化的机动能力。

## 七、第七层提醒：真正强的项目系统，不能只靠命令维持，还要靠“官兵一致、军民一致”式的关系质量

谈话里总结八路军政治工作时，提出“官兵一致、军民一致、宽待俘虏”。

翻到软件团队里，我觉得至少能转译出三条特别有用的原则。

第一，官兵一致：
不要让拍板的人和干活的人活在两个世界里。
如果管理者只会压指标，不理解真实负担；如果工程骨干长期吃苦，却没有位置影响判断，组织迟早会裂开。

第二，军民一致：
不要让项目团队和用户、一线支持面长期对立。
如果项目推进总是以牺牲客服、实施、用户体验和现场关系为代价，它很难形成真正稳的胜利。

第三，宽待俘虏：
关键项目中不必把所有过去不同意你的人、曾经反对过你的人都永久打成敌人。
更成熟的做法，是给愿意转向的人留下加入的空间，让更多原本摇摆的力量可以被吸纳进新结构。

真正成熟的组织，既有纪律，
也懂得把更多人转化成可合作力量。

## 八、第八层提醒：统一指挥很重要，但统一指挥不是把一切差异压平，而是让多种力量围绕主战场配合

谈话里除了谈广泛动员，也反复讲统一指挥。
这提醒软件团队一件很重要的事：
关键项目不能无指挥、无主次、无节奏。

但统一指挥并不等于每个团队都必须完全一样地做事。

现实里真正高质量的统一，往往是：
- 产品团队负责定义主战场和成功口径
- 工程团队负责打穿关键技术和交付链路
- 测试、SRE、数据、客服负责把一线真相带回来
- 设计、运营、销售等按不同角色协同主目标
- 各自打法不同，但方向、节奏和指挥统一

这很像文章里讲的：
大量军队与八路军打法不同，但要形成互相配合。

对软件团队来说，最糟糕的不是角色不同，
而是方向不同、时钟不同、胜利定义不同。

## 九、这篇谈话对今天软件团队的最大启示：关键项目不能再只是“少数人苦撑 + 全组织补洞”了

如果让我把《和英国记者贝特兰的谈话》对今天软件团队的启示压成几句话，我会这样总结：

第一，很多关键项目之所以越打越险，不是因为没人努力，而是它还只是管理层和少数骨干的战争，不是组织真正参与的战争。
第二，真正的广泛参与，不是更多人知道项目存在，而是更多人知道自己为什么进场、如何进场。
第三，阶段已经变了，旧流程、旧权限、旧协作方式不改，项目就会被老系统拖死。
第四，很多团队最大的失误，不是还不够拼，而是一直在打单纯防御，始终没有主动攻击决定性瓶颈。
第五，关键项目需要一种既让真实信号上来、又能把统一决断落下去的治理方式。
第六，不要把所有工作都做成笨重的正面阵地战，要保留有纪律的机动能力。
第七，真正稳的项目系统，不只靠命令维持，还要处理好管理层、承担者、用户和一线支持面之间的关系。

## 十、给今天软件团队的一句落地建议

如果把这篇谈话压成一句今天最好用的话，我会这样说：
别再让关键项目只靠几个 leader 和骨干工程师苦撑，也别再把全部精力都花在补洞和防守上。你真正该做的，是把测试、设计、运维、客服、销售、依赖团队和一线反馈都组织进主战场，改掉不适配新阶段的旧治理，并主动集中火力攻击那个最决定成败的瓶颈。很多项目不是坏在没人重视，而是坏在它始终没有从“少数人的防守战”升级成“全组织参与的主动战”。
