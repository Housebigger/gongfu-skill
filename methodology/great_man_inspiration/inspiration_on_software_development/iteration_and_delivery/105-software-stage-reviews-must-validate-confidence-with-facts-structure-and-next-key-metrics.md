# 从《三个月总结》看软件团队：阶段复盘不能只打气，要用事实、结构和下一阶段关键指标校验项目方向

日期：2026-05-15

原文对应：
《三个月总结》
原文链接：https://www.marxists.org/chinese/maozedong/marxist.org-chinese-mao-19461001.htm

写作背景：
1946年10月1日，毛泽东为中共中央起草《三个月总结》，对全国规模内战爆发后三个月的战争经验作系统复盘。文章不是泛泛鼓劲，而是用战果、敌我兵力结构、失地与收复前景、游击根据地建设、东北休整空间、集中优势兵力的战法、扩兵训练、财政生产、土地改革、地方武装、国统区群众斗争和国际国内政治形势等一整套事实，来校验“我们能够战胜蒋介石”的判断。它既承认斗争长期、残酷、会有损失，也明确提出下一阶段的关键任务：继续歼敌、转换战略态势、组织全党紧张动员，并把军事、生产、训练、土地改革、群众工作与政治解释统成一个整体推进。

说明：
转译到软件开发，105提醒团队：版本复盘、季度复盘、项目总结最怕变成情绪会、表态会和鼓劲会。真正有用的阶段复盘，必须回答几个硬问题：项目有没有更接近胜利，决定成败的结构矛盾是什么，下一阶段最关键的指标与主战场是什么，团队要把哪些资源重新组织起来。

核心思想的当代转译：
如果把105翻成软件团队语言，我会概括成一句话：阶段复盘不能只打气；要用交付事实校验信心，用系统结构判断下一步，再把资源压到能改变局势的关键指标上。

## 一、复盘先看交付事实，不要拿情绪代替证据

毛泽东用三个月实际歼敌结果证明判断正确。软件团队复盘也要先看事实：版本是否按期、故障是否下降、用户问题是否闭环、关键依赖是否清掉、技术债是否被真实削减。不能因为大家都很拼，就把复盘写成“总体向好”；也不能因为过程很痛苦，就忽略已经形成的真实改善。

## 二、要识别决定成败的结构矛盾，而不是沉迷表层 bug 列表

文章看到的是蒋军战线太广与兵力不足的结构矛盾。软件项目里同样如此：真正决定成败的，也许不是某几个孤立 bug，而是需求面太广、核心人力太少、平台依赖太重、测试与发布链路太弱。复盘若只停留在 issue 清单，就抓不住项目的胜负手。

## 三、下一阶段必须明确关键指标和主战场

105提出再歼敌二十五个旅这个关键量化目标。软件团队也要有类似表述：比如把 crash 率打到多少，把主链路延迟降到多少，把积压需求清到多少，把迁移完成到哪个里程碑。没有关键指标，团队的努力就会重新分散回杂乱任务里。

## 四、允许阶段性退让，但必须马上检讨打法并避免重犯

文章承认失地，要求各地检讨作战经验、吸取教训。软件项目里，这对应着：你可能丢掉一个版本窗口、回滚一次发布、延期一个里程碑，但复盘的重点不是自责，而是检讨为什么会丢、哪些是主动收缩、哪些是错误导致的、以后如何避免再犯。

## 五、交付、训练、补给、组织机制必须一起推进，不能只修代码不修系统

105把训练、生产、财政、土地改革、群众工作放在一个整体里。软件团队也要明白：项目改善不能只靠工程师多写一点代码，还要同步加强测试训练、发布纪律、环境建设、文档、协作机制、用户反馈回路和组织解释工作。真正的项目转折，来自系统性重组，而不是单点拼命。
