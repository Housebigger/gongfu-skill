# 从《大量吸收知识分子》看技术团队建设：很多团队并不是缺人，而是先把那些跨背景、跨领域、真正能带来升级的人当成“不合群风险”挡在门外

日期：2026-04-21

原文对应：
《大量吸收知识分子》

写作背景：
1939年12月，毛泽东写下《大量吸收知识分子》，专门纠正党内和部队中对知识分子的排斥心理。那时不少干部一听“知识分子”四个字，第一反应不是如何用其所长，而是先担心他们不能吃苦、不够稳定、不像自己人、不好领导。毛泽东非常明确地指出：没有知识分子的参加，革命的胜利是不可能的。更重要的是，这篇文章并不只是喊一句“要重视人才”，而是进一步分析了排斥是怎么形成的、应该吸收什么样的人、怎样甄别、怎样教育、怎样让新旧力量结合起来。放到今天的软件开发里，这几乎就是一篇关于技术团队如何避免“纯血论”、如何接住异质人才、如何把不同背景的人真正变成团队能力的经典方法论。

说明：
今天把这篇文章转译到技术团队语境里，当然不能机械照搬战争年代的政治表述。但它对现代软件组织极有现实感。很多团队嘴上都说工程师难招、靠谱的人太少、业务复杂、交付压力大，可往深处看，很多团队真正的问题不只是“没有人”，而是内部天然带着一种排斥机制：只信任和自己技术路径相似的人，只欢迎表达方式熟悉的人，只愿意接纳不会冲击现有习惯的人。凡是跨栈、跨角色、跨行业、跨文化、表达更锋利、方法更系统的人，常常先被贴上“不懂我们业务”“不适合团队风格”“写法太不一样”“合作成本太高”的标签。结果就是，团队一边喊缺人，一边又把最可能补短板、带来新方法、提升工程质量的人亲手推走。

核心思想的当代转译：
如果把《大量吸收知识分子》翻成今天技术团队建设的语言，我会把它概括成一句话：
很多技术团队并不是缺人，而是先把那些跨背景、跨领域、真正能带来升级的人当成“不合群风险”挡在门外。成熟的工程团队，不是只找“像现有工程师的人”，而是做到五件事：开放吸收、扎实甄别、清晰安放、组织性带教、让新旧能力一起升级。

## 一、第一层提醒：技术团队最常见的人才误判，不是看不见候选人，而是天然更相信“像自己的人”

很多技术团队一谈到招人，
最常见的抱怨是：
- 现在市场上靠谱工程师太少
- 真正能写的人不好找
- 候选人简历看起来都差不多
- 业务太复杂，一般人很难快速上手

这些判断当然不是全错。

但如果再往深里看，
很多团队真正的问题，
并不只是人才稀缺，
而是接纳能力很窄。

因为一旦出现一个有潜力的人，
只要他带着下面这些特征中的一两个：
- 不是传统科班路径
- 之前做的领域和当前业务不完全一样
- 来自产品、设计、测试、数据、运维等跨角色背景
- 技术表达更直接
- 对旧代码和旧流程会提出尖锐问题
- 使用的工具链、工程习惯和团队现有方式不同

很多团队的第一反应往往不是：
“这个人也许能补我们的盲区。”
而是：
- “他不一定适合我们。”
- “他写法跟我们不一样。”
- “他这种背景转过来靠不靠谱？”
- “他会不会不懂真正的工程现场？”
- “他来了会不会把团队节奏搞乱？”

于是问题就来了。

表面上团队在招聘，
实际上却只愿意接受一种很窄的人：
- 技术路径和现有成员高度相似
- 表达方式熟悉
- 对现有习惯天然服从
- 不会明显改变团队认知结构的人

但真正能带来团队升级的，
很多时候恰恰不是“重复你的人”，
而是“补你的人”。

如果一个工程团队核心成员：
- 技术路线高度同质
- 问题拆解方式高度同质
- 对质量的理解高度同质
- 角色视角高度同质
- 对用户和业务的理解长期封闭在单一圈层里

那么它短期看起来也许磨合成本低，
长期却很容易越来越盲。

因为你看到的问题、
想到的方案、
愿意讨论的边界，
都会被这套同质化结构悄悄锁住。

所以《大量吸收知识分子》给技术团队的第一个提醒就是：
很多团队招不到关键人才，
不是因为完全没人，
而是因为自己只愿意相信“像自己的人”。

## 二、第二层提醒：技术团队缺的常常不是单纯的人头，而是能补能力结构的新力量

毛泽东在原文里反复强调：
革命要胜利，
不能没有知识分子的参加。

翻成今天的软件开发语言，
这句话的含义非常直接：
任何想做复杂产品、
想提高工程质量、
想支撑更大规模系统、
想从草台交付走向真正工程化的团队，
都不能只靠原有经验路径硬撑，
一定需要吸收新的能力结构。

这里说的“新力量”，
不只是多招几个人，
而是那些能让团队进入下一阶段的人：
- 更系统地理解架构和边界的人
- 更重视测试、稳定性和可维护性的人
- 更懂产品与用户语境的人
- 更能做文档化、流程化和机制化沉淀的人
- 更有跨团队协作经验的人
- 更能把模糊问题讲清楚、拆清楚、落清楚的人

很多技术团队在早期之所以能跑起来，
靠的是：
- 几个核心工程师的高强度投入
- 创始成员之间的默契
- 大量隐性知识和口口相传
- 先把功能做出来再说的冲劲

这些都非常重要。

但一旦产品和组织开始进入更复杂阶段，
比如：
- 模块变多
- 协作角色变多
- 上线频率更高
- 故障影响面更大
- 用户反馈链路更长
- 质量问题开始系统性暴露
- 技术债开始反过来吞噬交付速度

原来那套“靠熟人、靠经验、靠少数骨干硬扛”的方式，
就越来越不够。

这时候团队如果还排斥：
- 跨领域转入的人
- 更注重流程和质量的人
- 更懂协作机制的人
- 更有业务理解或用户视角的人
- 对既有做法提出质疑的人

本质上不是在守护团队，
而是在拒绝升级。

所以技术团队说自己“缺人”，
很多时候真正缺的，
不是更多相同的人，
而是能补现有能力结构的新力量。

## 三、第三层提醒：真正成熟的招人方法，不是“纯血筛选”，而是“开放吸收 + 扎实甄别”

很多技术团队排斥异质人才，
最常见的理由听上去都很专业：
- 我们不是不要他，是他技术栈不完全匹配
- 我们不是保守，是这个岗位容错率低
- 我们不是排斥，是工程标准必须守住
- 我们不是只招自己人，是文化契合很重要

这些理由不是一点道理都没有。

工程团队确实需要标准，
岗位也确实有试错成本。

问题在于，
很多团队最后把“标准”用成了排异工具：
因为怕有风险，
所以只敢招最熟悉的人；
因为怕沟通成本高，
所以只愿意招表达方式和自己接近的人；
因为怕上手慢，
所以默认跨背景候选人都不值得投入。

于是团队就很容易形成一种危险的招聘逻辑：
- 表面上强调高标准
- 实际上越来越保守
- 表面上说追求契合
- 实际上是在筛同类
- 表面上说对质量负责
- 实际上是把升级机会挡在门外

毛泽东在原文里给出的路子，
恰恰不是两种极端里的任何一种：
不是谁都收，
也不是因为担心混入不合适的人就整体关门，
而是同时坚持两件事：
一是大胆吸收，
二是具体甄别。

翻到今天的软件团队里，
这意味着什么？

意味着你的招聘方法应该是：
- 候选池要开放，而不是只在同温层里转
- 技术判断要扎实，而不是靠熟悉感替代判断
- 接受合理范围的试错，而不是追求零风险招人
- 看核心能力，而不是过度迷信履历同质性

真正该看的，
应该是：
- 问题分析能力是否过关
- 基础技术素养是否扎实
- 学习速度是否足够快
- 工程习惯是否可靠
- 面对真实约束时是否有建设性
- 协作方式是提升系统还是破坏系统

如果这些核心项成立，
那么很多“和我们不太一样”的差异，
本来就不该成为否决理由。

一个团队最危险的，
不是曾经招进过几个不合适的人，
而是被少数坏经历吓住之后，
从此对真正有潜力的异质人才也整体关门。

## 四、第四层提醒：新人能不能融进来，很多时候不是新人的问题，而是团队有没有承担组织性带教责任

很多技术团队都会说：
“我们其实并不排外，
只是有些新人自己融不进来。”

这句话听起来很自然，
但很多时候并不公平。

因为“融不进来”这件事，
从来不只是新人个人的问题，
更是团队设计、协作接口、文档质量、反馈机制、带教安排共同作用的结果。

毛泽东在原文里并不只讲“吸收”，
同时强调“教育”和“团结”。

翻成今天软件团队的话，
就是：
把人招进来，
并不等于任务完成；
真正的组织任务，
是让他能进入真实代码现场、
理解业务上下文、
建立协作关系、
把他的长处变成团队能力，
而不是让他一个人去适应一切。

现实中很多所谓“新人不行”的案例，
根本不是人不行，
而是团队没接住。

常见表现非常典型：
- 入职文档不完整，却要求新人一周内进入状态
- 代码规范靠“大家都知道”，没人讲清楚
- PR 反馈只会挑错，不会解释背后的判断框架
- 老成员默认新人“先自己摸”，出了问题再统一归因为能力差
- 业务上下文全在少数人口头里，别人接触不到
- 跨团队协作接口混乱，却把沟通失败归因给新人不会配合

这样一来，
再有潜力的人也很容易被组织耗掉。

所以一个团队如果真想用好多样背景的人，
至少要承担五个责任：
- 责任一：把关键上下文写出来，而不是默认为“大家应该懂”
- 责任二：给清晰的角色边界和前期任务，不让新人在模糊地带耗死
- 责任三：把 code review 变成带教，而不是单纯审判
- 责任四：在新旧冲突出现时做翻译和仲裁，而不是默认旧人正确
- 责任五：把新人的优势嵌进正式机制，而不是让它永远停留在个人层面

真正成熟的工程团队，
不会把 onboarding 理解成“你自己尽快适应”，
而会把它理解成：
组织必须主动搭桥、解释、设计路径、缩短无意义磨损。

## 五、第五层提醒：技术团队引入异质人才，不只是补人手，而是逼整个团队升级

很多团队在吸收新成员时，
潜台词其实是：
“你可以来，
但最好完全按我们现在的方式来。”

这就很容易把“引入人才”做成“同化人才”。

看上去团队扩充了，
实际上工程能力并没有升级。

毛泽东在原文里讲得很深的一点，
是他不只要求知识分子向工农干部学习，
也要求工农干部提高文化水平。

这背后的方法论非常先进：
真正的结合，
从来不是一边改、一边不动；
而是双方都在结合中成长。

放到今天的软件团队里，
这个提醒尤其重要。

因为很多老成员、老代码、老习惯，
天然占据“正统”位置：
- 我们一直就是这么写的
- 这个系统历史包袱大，你别想得太理想
- 先把现有做法学会再谈改进
- 别一来就挑战老系统

这些话不是完全不能说。

问题在于，
如果它们最后变成了拒绝更新的盾牌，
那么团队即使引入了更有方法的人、
更懂测试的人、
更懂协作的人、
更懂产品的人，
也依然不会真正变强。

真正有远见的技术负责人会明白：
吸收异质人才，
本来就意味着老团队的一部分成员要被迫升级。

这种升级可能体现为：
- 以前靠默契推进的地方，要学会文档化
- 以前靠“能跑就行”的地方，要接受更严格的质量讨论
- 以前只从代码角度看问题的人，要学会补上产品和用户视角
- 以前拍脑袋定边界的方式，要转向更可解释的工程判断
- 以前“我懂就行”的个人英雄模式，要过渡到团队可复制的工作方式

这当然会让一些人不舒服。

但工程团队升级，
本来就不可能在所有人都毫无不适的情况下发生。

所以《大量吸收知识分子》给软件团队的第五层提醒就是：
你引入新的人，
不是为了多一双手，
而是为了逼整个系统提升能力密度。
如果老系统完全不需要变化，
那通常说明这次“吸收”只停留在形式上。

## 六、第六层提醒：技术团队真正挡住好人的，往往不是面试题难度，而是那股隐形的“纯血论”氛围

很多团队复盘招人失败时，
第一反应往往是：
- 薪资竞争不过大厂
- 候选人水平不稳定
- 行业太卷
- 业务复杂度太高

这些当然都是现实因素。

但现实里还有一种很容易被低估的力量，
就是团队内部那股隐形的“纯血论”氛围。

什么叫“纯血论”？
不是公开写着“只欢迎某一种人”，
而是日常里到处透出的信号：
- 不是我们这套技术路径出来的，就默认要被多怀疑一层
- 不按现有风格说话的人，就容易被认为沟通有问题
- 更重视产品和用户的人，会被说“不够工程”
- 更强调流程、测试和稳定性的人，会被说“速度太慢”
- 来自非传统背景的人，必须比别人多证明很多次

这些东西单独看都像小事，
合在一起就会形成一个很强的筛选器：
真正有能力、
有方法、
有不同视角、
有升级潜力的人，
会迅速感觉到这不是一个真正欢迎变化的团队。

于是他们要么不来，
要么来了也很快撤退。

最后留下来的，
往往更多是：
- 容易迎合既有风格的人
- 不会挑战团队舒适区的人
- 可以在低标准下稳定共处的人

这时团队表面上看“氛围很好”，
实际上只是因为真正会带来变化的人都被筛走了。

所以一个技术负责人如果真想提高团队质量，
不能只盯着招聘漏斗，
还必须认真审视：
- 团队是不是天然更喜欢“像我们的人”
- code review 是否夹带路径歧视和语气排斥
- 面试里的“文化契合”是否实际上是在筛同类
- 老成员是不是把新人视为威胁而不是增量
- 不同背景的人进入后，是否真的能获得平等的专业尊重

薪资能把人拉到面前，
氛围才决定人能不能留下并发挥价值。

## 七、第七层提醒：技术人才市场本来就是竞争场，你不要的人，别的团队会要

毛泽东在原文里有一个非常现实的判断：
别人也在争夺知识分子。

这句话放到今天的软件行业，
几乎不需要解释。

任何真正有潜力的人，
都不可能只被你看到。

你觉得他：
- 学得快
- 想得清
- 写得稳
- 能补团队短板
- 能带来不同视角
- 能在未来长成骨干

别的团队大概率也会这样判断。

所以技术团队对异质人才的态度，
从来不只是一个内部管理问题，
同时也是一个外部竞争问题。

这意味着什么？

意味着你的排斥病，
不是一种中性的团队习惯，
而是一种真实的竞争劣势。

因为当你还在想：
- “他是不是不够像我们？”
- “这种背景会不会不稳定？”
- “他会不会把协作搞复杂？”
- “要不要再等等更标准的人？”

别的团队可能已经在做另一件事：
- 更快识别他的价值
- 更清楚地定义他的角色
- 给出更好的带教和接口
- 让他更早进入关键问题现场

于是最后的结果就是：
你用排斥保护了熟悉感，
别人用开放拿走了未来能力。

很多团队不是输在后来学不会，
而是输在前面不敢要。

所以从竞争角度看，
大胆吸收异质人才，
本身就是一种工程战略动作。

它决定的不是今天多招一个人，
而是未来几年你的团队有没有可能形成新的能力密度。

## 八、落到团队实践上，真正可执行的方法，不是“口头包容”，而是“吸收、甄别、安放、带教、共长”五步连起来

如果把这篇文章彻底翻成今天的软件团队动作，
我会把它落成五步法。

### 1. 吸收：先打掉“不是自己人”的默认防御
团队管理者要先在认知上过关：
不要把跨背景、不同表达方式、不同技术路径、不同角色经验，
自动理解成风险。

很多团队的第一道门槛，
不是候选人过不去，
而是面试官和老成员的偏见过不去。

### 2. 甄别：看核心项，而不是看熟悉感
真正该看的，
是问题分析能力、工程习惯、学习速度、协作方式、质量判断和价值底线。

不要把“像不像我们现在的人”、
“是不是按我们的方式说话”、
“会不会让我不舒服”
当成最关键的标准。

### 3. 安放：把人放对位置，比招进来更重要
很多人才不是没价值，
而是被塞进了一个既不能发挥长处、又容易放大短处的位置。

团队应该让人的长板进入真正需要他的地方：
- 谁适合基础设施
- 谁适合业务复杂区
- 谁适合质量治理
- 谁适合产品协同更强的位置
- 谁适合把零散经验整理成机制

如果只会机械分工，
很多潜力都会被白白浪费。

### 4. 带教：让 onboarding 成为组织能力，而不是个人自生自灭
真正有效的带教，
应该包括：
- 环境和上下文的清晰交接
- 关键模块的结构化介绍
- 代码风格背后原因的解释
- review 时既指出问题也说明判断框架
- 在前几次真实任务中提供足够支撑

如果这些都没有，
那很多所谓“新人不适合”，
本质上只是组织懒惰的后果。

### 5. 共长：让新人的进入，变成整个团队的能力抬升
最终目标，
不是让新人“学会像老人一样做事”，
而是让整个团队因为这次吸收而变得：
- 更会解释
- 更会协作
- 更会复盘
- 更会做质量治理
- 更能从不同视角看系统问题

如果一个团队每次招人都只是补坑，
却从来没有因此变得更成熟、
更专业、
更可复制，
那说明它其实还没真正学会吸收人才。

## 九、这个逻辑不只适用于工程师招聘，也适用于团队如何对待产品、设计、测试、运维等“非纯开发”视角

这篇文章给技术团队还有一个很重要的延伸提醒。

很多工程团队排斥异质开发者的逻辑，
和排斥产品、设计、测试、运维、数据视角的逻辑，
本质上是同一个病根。

比如很多团队天然更欢迎：
- 写代码像自己的人
- 说话像自己的人
- 思考边界和自己接近的人

而对那些：
- 会更频繁从用户视角提问题的人
- 会对交互和体验提出质疑的人
- 会反复追问测试和稳定性的人
- 会强调交付链路和运维约束的人
- 会要求文档、流程和接口清晰化的人

则更容易先入为主地觉得：
“这些人不是真正做技术的。”

结果就是，
团队一边抱怨返工多、质量差、跨团队沟通痛苦，
一边又把最能帮助自己补齐这些问题的视角挡在门外。

这并不是说团队不要守住技术标准。

而是说，
真正成熟的技术组织，
不是只会使用“最像传统工程师的人”，
而是能够在守住工程底线的前提下，
把那些原本看起来“不够典型”、
但真实能补结构短板的人和视角吸收进来。

## 十、最后回到一句最朴素但最重要的话：团队别总说缺人，先看看是不是自己先把人推走了

《大量吸收知识分子》最值得今天技术团队反复咀嚼的，
不是一句浅层的“重视多元人才”，
而是它逼管理者正视一个更难听、但更真实的问题：
很多团队的问题，
不是缺人，
而是排人。

缺人是一种外部困难，
排人是一种内部病。

外部困难需要时间解决，
内部病如果不治，
就算偶尔招来了好的人，
最后也会被系统慢慢吐出去。

所以真正成熟的技术团队建设，
不该停留在：
- 我们也欢迎优秀人才
- 我们当然重视多元背景
- 我们不是排外，只是标准高

而应该更进一步追问：
- 我们是不是只欢迎“像现有工程师的人”
- 我们有没有用文化契合包装路径歧视
- 我们有没有把组织带教的缺位，包装成对新人的高标准
- 我们有没有真的承担起融合和升级的责任
- 我们是不是一边喊招不到人，一边不断把真正能补团队的人推出去

如果这些问题不面对，
团队就会长期卡在一种假性缺人状态里：
表面上持续招聘，
实际上却没有真正建设起一个能持续吸收、转化和放大人才的工程系统。

而这篇文章给今天软件团队最有力量的提醒，
恰恰就是：
真正强的团队，
不是靠一小撮路径相同的人永远硬扛出来的，
而是敢于持续吸收不同背景的人、接住不同类型的能力、
把差异变成视角优势、
把带教变成组织责任、
把融合变成系统升级，
最终把团队建设真正做成产品与工程质量的核心基础设施。

一句话收束：
技术团队真正高水平的建设，不是守着熟悉感过日子，而是有能力把“原本不像自己人、但真正有价值的人”接进来、用起来、带出来，并最终让整个团队因此更强。
