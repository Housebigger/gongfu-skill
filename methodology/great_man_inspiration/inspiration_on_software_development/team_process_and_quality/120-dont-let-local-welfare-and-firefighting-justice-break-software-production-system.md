# 从《关于工商业政策》看软件团队治理：别让局部福利和救火正义破坏生产系统

日期：2026-05-20

原文对应：
《关于工商业政策》
原文链接：https://www.marxists.org/chinese/maozedong/marxist.org-chinese-mao-19480227.htm

写作背景：
1948年2月27日，毛泽东为中共中央起草《关于工商业政策》。文章针对的是某些地方在接管和治理过程中，违反党中央关于工商业的政策，实际破坏工商业的问题。它从两个层面检查错误：一是领导方针上，不能把农村中斗争地主富农、消灭封建势力的方法，错误地用于城市工商业；要严格区别消灭封建剥削和保护地主富农所经营的工商业；要坚持发展生产、繁荣经济、公私兼顾、劳资两利，而不是用片面狭隘的救济方针破坏工商业。二是领导方法上，上级必须通过通信、报纸、谈话等方式密切掌握执行过程，随时交流经验、纠正错误、划清许做和不许做的界限，而不能等数月、半年、一年后才总纠正。

说明：
今天把这篇文章转译到软件团队治理，当然不能照搬当年的政治和经济语境。但它对软件组织有一个非常硬的提醒：团队不能把一种场景下的硬治理搬到所有场景；也不能为了局部舒适、局部正义、短期止痛，破坏整个产品生产系统。软件团队的福利、效率、质量和交付，不是彼此孤立的口号，而是同一个系统里的变量。

核心思想的当代转译：
如果把《关于工商业政策》翻成今天软件团队的语言，可以概括成一句话：不要用救火式、情绪式、局部最优式的管理动作，打坏产品研发的生产链；真正好的团队治理，是在明确政策边界的前提下，让工程师、产品、测试、运维、业务和客户支持一起降低返工成本、提高交付质量、改善协作体验，并在执行中实时纠偏。

## 一、第一层提醒：别把事故处理办法，当成日常治理办法

文章批评的第一类错误，是把一个领域里的斗争办法搬到另一个领域。软件团队里也常见这种误用。

线上事故时，强管控、冻结发布、集中审批、统一指挥，可能是必要的。但如果把事故状态下的管理方式长期搬到日常研发，团队就会窒息。反过来，探索期产品需要宽松试错，但如果把探索期的松散方式搬到稳定业务系统，也会制造质量灾难。

很多团队的问题，不是没有制度，而是制度不分场景：所有项目都按 P0 事故管，所有需求都按创新项目放，所有人都按不可信对象审批，所有线上变更都靠英雄值守兜底。结果就是，管理看似严格，实际生产能力下降；流程看似保护质量，实际只是把责任推远。

成熟的软件治理，第一步是区分场景：探索、交付、运维、事故、重构、迁移，各有不同方法。

## 二、第二层提醒：别用“关心工程师”破坏工程生产系统

文章特别区分了发展生产、繁荣经济、公私兼顾、劳资两利的正确方针，和那种表面拥护工人福利、实际破坏工商业的狭隘救济方针。软件团队里也有类似陷阱。

比如，为了减少工程师压力，直接取消所有文档、测试、评审、值班、复盘，看似照顾大家，实际会让线上问题更多，返工更多，救火更多。又比如，为了显示尊重研发，拒绝任何排期约束和业务承诺，看似保护工程师，实际会让产品信用下降，客户关系受损，最后团队预算和岗位都不稳。

真正的工程师福利，不是把生产纪律全部拆掉，而是减少无意义返工、减少反复插队、减少需求摇摆、减少靠人肉记忆维持系统的痛苦。好的福利要和生产系统一起设计，而不是站在生产系统的反面。

## 三、第三层提醒：建立跨角色的“生产管理委员会”

原文提出，在政府领导下，工人与资本家可以组织生产管理委员会，降低成本、增加生产、便利推销。转译到软件团队，就是要把工程、产品、测试、SRE、设计、销售、客户成功和运营拉进同一个生产改进闭环。

很多软件团队的问题，是每个角色都只优化自己局部：产品只要需求快进，工程只要代码边界清楚，测试只要入口稳定，SRE只要少变更，销售只要尽快承诺，客户成功只要客户别闹。局部看都合理，合起来却是系统性混乱。

真正有效的机制，是共同看几件事：需求从哪里来，变更怎样进入，优先级谁来裁决，测试如何前置，发布如何降风险，事故如何复盘，客户反馈如何进入产品，技术债如何进入排期。团队治理不能只在“谁更辛苦”上打转，还要回到“怎样让生产系统更顺”。

## 四、第四层提醒：政策边界必须提前讲清，不能等事故扩大后再总纠正

文章在领导方法上很强调实时联系和及时纠偏。软件团队也一样。很多流程改革失败，并不是因为方向全错，而是因为边界没有讲清，执行中也没人看。

代码评审到底是质量协作，还是权力卡口？技术债治理到底优先处理什么，不处理什么？线上事故复盘到底是找责任人，还是找系统漏洞？值班制度到底保护谁，约束谁？需求插队到底什么情况下允许，谁有最终裁决权？

如果这些不讲清，团队就会用各自理解执行同一个政策。几个月后管理层才发现：有人把评审当官僚审批，有人把敏捷当不要计划，有人把自驱当无限加班，有人把质量门禁当甩锅工具。到那时再纠正，信任已经损耗很多。

## 给今天的软件实践提醒

第一，事故场景、探索场景、稳定交付场景，不能套同一套治理动作。

第二，真正的工程师福利，不是取消生产纪律，而是减少返工、混乱、救火和无效消耗。

第三，局部最优经常会破坏系统最优，团队要建立跨角色的生产改进闭环。

第四，任何流程改革都要提前讲清：什么可以做，什么不可以做，谁来裁决，如何复盘。

第五，不要等季度复盘才纠偏。软件生产系统的错误，越晚发现，扩散成本越高。
