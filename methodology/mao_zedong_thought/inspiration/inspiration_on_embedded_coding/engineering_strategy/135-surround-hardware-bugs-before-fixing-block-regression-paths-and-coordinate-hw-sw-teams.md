# 从《关于平津战役的作战方针》看嵌入式开发：先围后打，堵住回归路径，硬软件团队协调出击

- 日期：2026-06-12
- 原文对应：《关于平津战役的作战方针》（1948 年 12 月 11 日，毛泽东为中央军委起草的电报）
- 原文链接：https://www.marxists.org/chinese/maozedong/marxist.org-chinese-mao-19481211.htm
- 写作背景：毛泽东在平津战役中提出"围而不打、隔而不围"——先完成对敌全面包围部署，堵住海上逃跑通道，等部署完成再逐一歼灭。全局协调甚至让淮海战场也配合延缓进攻。
- 说明：本文把"先围后打、堵退路、全局协调"的包围艺术，转译为嵌入式系统开发的工程实战方法论。
- 核心思想当代转译：解决复杂硬件/固件 bug 时不要急于上电调试——先完成"包围"（搭建观测手段、隔离变量、准备回滚方案），堵住"退路"（锁定版本、冻结变更），等一切就绪再动手定位。

---

## 一、先围后打——先搭好观测手段再定位 bug

毛泽东要求两周内"围而不打"——先完成包围部署，不急于歼灭。

- **急于"打"的代价**：一发现 bug 就疯狂改代码试——改了 A 出了 B，改了 B 出了 C，最后不知道改了什么、原始状态也回不去了
- **先围后打的做法**：先搭好完整的日志输出（围住行为），先用示波器/逻辑分析仪确认信号（围住硬件），先用单元测试复现问题（围住触发条件），等这些观测手段就位了再定位根因
- **围 = 创造可观测性**：看不见的 bug 打不了——先让 bug 的行为完全暴露在你的观测范围内，再动手修

> 135 提醒：如果你反复修一个 bug 却总修不好，问题不是水平不够，而是你没有先完成"包围"——没有让 bug 的行为完全暴露在观测范围内就动手了。

## 二、堵住回归退路——锁定版本比急着修更重要

毛泽东说"唯一怕敌人从海上逃跑"——塘沽海口是关键。

- **嵌入式中的"海上逃跑"**：
  - 修了一个 bug 引入两个新 bug——敌人从海上跑了又从别的地方冒出来
  - 没有版本锁定——改着改着不知道当前跑的是哪个版本
  - 没有回归测试——修完不知道有没有破坏其他功能
- **堵退路的方法**：
  - 版本控制：每次改动前 git commit，确保随时可以回退到已知状态
  - 自动化回归：CI 跑全量测试，修完立刻知道有没有引入新问题
  - 变更冻结：定位 bug 期间不允许其他人改动相关模块

> 135 提醒：修 bug 最怕的不是修不好，而是越修越多。先堵住回归退路（版本锁定+回归测试），再动手修。

## 三、硬件、软件、固件团队协调——全局一盘棋

毛泽东让淮海战场配合平津——多战场全局协调。

- **嵌入式多战场**：硬件设计、固件开发、驱动适配、应用层——不是独立的，必须协调
- **协调的例子**：
  - 定位一个时序 bug：硬件团队用示波器确认信号波形（淮海战场），固件团队同步加日志（平津战场），两边信息交叉确认才能定位根因
  - 不协调的后果：硬件团队改了电路但没通知固件团队（淮海打太快），固件按旧时序配置跑飞（平津被误伤）
- **协调的方法**：统一的 bug 跟踪系统、每日站会同步进展、共享的测试报告

> 135 提醒：嵌入式 bug 往往跨硬件和软件边界。别各修各的——先对齐信息，再分工定位。

## 四、不要过分估计 bug 的复杂度——可能是惊弓之鸟

毛泽东说"都已成惊弓之鸟，切不可过分估计敌人的战斗力"。

- **嵌入式中的惊弓之鸟**：一个看似复杂的时序 bug，可能只是一个上拉电阻选错了值——当你把观测手段做到位（围住了），根因往往比你想象的简单
- **常见的过高估计**：
  - "这个死机问题肯定是 CPU 有问题"——最后发现是电源纹波超标
  - "通信协议太复杂肯定实现有错"——最后发现是时钟配置差了 1%
  - "这个中断延迟肯定跟 RTOS 调度有关"——最后发现是中断优先级配置反了
- **为什么容易高估**：因为没有足够的观测手段，只能猜测——猜测总是往最坏方向猜

> 135 提醒：先用观测手段把 bug 行为完全暴露出来，再判断复杂度。多数 bug 在充分观测后都是惊弓之鸟。

## 五、攻击次序有讲究——先堵关键路径

毛泽东排了严格次序：第一塘芦（海口），第二新保安……最后北平。

- **嵌入式 debug 次序**：
  - 第一步：确认电源和时钟（塘沽——堵退路，所有问题的前提条件）
  - 第二步：确认硬件信号链路（新保安——验证硬件通路）
  - 第三步：确认固件基础配置（唐山——寄存器、时钟树、引脚复用）
  - 第四步：确认驱动和应用层（天津/张家口——上层逻辑）
  - 最后：性能优化和边界 case（北平——锦上添花）
- **为什么电源排第一**：电源不稳定的话，所有上层调试都是在沙子上建房子——塘沽不先拿下来，其他打了也白打

> 135 提醒：debug 有严格次序——先确认基础条件（电源、时钟），再逐层往上排查。跳过底层直接调上层是最常见的浪费时间行为。

## 六、复用已验证的 debug 方法——不重新发明轮子

毛泽东说"大体上即是用过的办法"——复用辽沈经验。

- **复用的价值**：上次调 I2C 时序用的方法（逻辑分析仪抓波形→对比协议→定位问题），这次调 SPI 时序直接用同样的流程
- **建立 debug 清单**：每次解决完一个复杂 bug，把"包围部署"记录下来——先做了什么、后做了什么、关键转折点在哪——下次遇到类似问题直接按清单走
- **不复用的代价**：每次都从零开始猜测，浪费大量时间在已经验证过的无效路径上

> 135 提醒：把你解决过的每一个复杂 bug 的 debug 路径记录下来。下次遇到类似问题，复用流程而非重新摸索。

---

## 结语

毛泽东这篇电报的核心方法论：先围后打，堵退路比攻城更重要；多团队全局协调；攻击次序有讲究；复用已验证的方法。

给今天嵌入式工程师的一句落地建议：**下一次遇到棘手 bug 时，不要急着改代码（打北平）——先搭好完整观测手段（围住），锁定版本和回归测试（堵退路），从电源和时钟开始逐层排查（先打塘沽），等 bug 行为完全暴露在观测范围内再动手修。围好了，修起来比想的容易得多。**
