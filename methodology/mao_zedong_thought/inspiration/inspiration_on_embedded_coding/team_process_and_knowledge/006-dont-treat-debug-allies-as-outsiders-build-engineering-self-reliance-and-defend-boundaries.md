# 从《和中央社、扫荡报、新民报三记者的谈话》看嵌入式团队：不要把一起扛板级问题的人先当异己，要把工程自力更生、联合调试和边界自卫做成下一阶段准备反攻的结构

日期：2026-04-18

原文对应：
《和中央社、扫荡报、新民报三记者的谈话》

写作背景：
1939年9月16日，毛泽东围绕抗战相持阶段、民主政治、党派磨擦、统一战线和国共合作等问题答记者问。这篇文章在“新阶段是准备反攻期”的判断之上，把方法进一步落到了阶段任务拆解、内部治理、自力更生、友党关系和严格自卫上。核心不是一句“要团结”，而是：第一，现在还不是立即战略总攻，而是分战线准备；第二，自力更生的主要内容不是硬扛，而是通过内部制度与政治改革增强抵抗力；第三，共同抗敌者不能先按“异党”处理；第四，对内部磨擦既反对对消力量，也坚持严格自卫；第五，真正的统一，只能统一于抗战、团结、进步。把它翻到嵌入式软件工程里，会变成一套很具体的团队协作与联合调试方法。

说明：
今天把这篇文章转译到嵌入式软件开发语境里，当然不能照搬战争年代的政治语言。现代嵌入式团队更常遇到的，是另一类“相持阶段”：一轮最凶的现场事故暂时止住了，板子不再天天挂死，量产风险没有继续扩大，版本虽然还远不稳定，但至少从“全面被动挨打”走进了“还没赢、但必须准备下一阶段”的阶段。这个时候，团队最容易死在三件事上：第一，把“准备反攻”误成“现在马上全线重构”或者“终于可以先不折腾”；第二，嘴上说靠自己，实际上日志、现场信息、硬件修订差异、责任边界、决策机制仍然混乱；第三，把本来应该一起扛系统问题的人——硬件、驱动、BSP、测试、FAE、工厂、供应商接口人——先当成互相限制的异己，最后问题没被设备打死，先被工程内耗拖垮。

这篇文章对今天嵌入式团队最值钱的一层，是它提醒我们：
真正的准备期，不只是继续熬夜和追 bug，而是把工程系统整成更会承压、更会联调、更会守边界的结构。真正的自力更生，不是让少数救火英雄继续燃烧，而是把团队升级成一个更能共同准备反攻的系统。

核心思想的当代转译：
如果把这篇文章翻成今天嵌入式团队的语言，我会把它概括成一句话：
当一个嵌入式项目已经从最危险的全面被动期走进“还没赢、但必须准备下一阶段”的阶段时，最重要的不是立刻全面重构，也不是先回到舒服协作，而是先把准备反攻的结构搭起来——把工程内部治理整清楚，把真正并肩扛板级问题的人按友军逻辑组织起来，把联调磨擦和接口越界纳入严格自卫规则，让整个系统统一在主问题、联合调试和持续进步上。

## 一、第一层提醒：准备反攻，不等于现在就全面重构；嵌入式准备期必须按战线拆任务

原文最重要的一层判断是：
准备反攻不是立即反攻。
战略层的大举反攻还没到，
但局部反击、防御巩固、后方改革都必须同时做。

翻到嵌入式项目里，这几乎就是现场常识。
很多项目刚把最危险的一轮问题压住，
就容易立刻走向两个极端：
- 要么说“终于稳一点了，先别动了”
- 要么说“现在马上全面重构、全面换栈、全面改框架”

这两种都可能重新把项目打回最坏阶段。

真正成熟的阶段判断是：
先承认现在还是准备期，
再把工程战线拆开。

例如：
- 现场战线：哪些现网/现地问题必须继续局部反击
- 稳定性战线：哪些核心指标必须先守住
- 架构战线：哪些长期问题要开始准备，但不能现在一把梭
- 生产战线：哪些量产、返修、工装、测试约束必须同步考虑
- 组织战线：哪些协作机制如果不改，后面永远打不出反攻能力

嵌入式项目特别怕一件事：
把“系统稍微不那么坏了”误成“现在可以随便动所有东西”。

## 二、第二层提醒：嵌入式团队的“自力更生”，不是让几个硬核工程师继续死扛，而是把工程内部治理做成真实抵抗力

原文里一句特别值得嵌入式团队反复咀嚼：
“自力更生的主要内容，就是民主政治。”

翻到工程里，它的意思不是照搬概念，
而是提醒我们：
一个团队真正靠自己，
不是靠少数懂系统的人继续熬，
而是靠内部治理升级。

在嵌入式项目里，这种治理升级具体包括：
- 现场信息能回到研发，不被中途截断
- 不同硬件修订、环境条件、复现路径能被结构化记录
- 驱动、应用、硬件、测试、工厂之间的责任边界更清楚
- 调试证据能被共享，而不是锁在某个人终端里
- 关键决策不是“谁声音大听谁”，而是基于事实闭环
- 团队能让更多人进入真实问题，而不是永远靠一两个神仙救火

很多嵌入式团队嘴上说“只能靠自己把问题顶住”，
实际上却是：
- 现场日志拿不全
- 复现条件没人统一
- 版本和板卡信息乱飞
- 问题一多就开始互相甩锅
- 真正懂系统的人被无限透支

这不叫真正的自力更生，
这叫把整个工程系统压成单点英雄模式。

真正成熟的嵌入式自力更生，
是让系统从“少数人硬顶住”升级成“更多人能进入同一事实面、同一调试闭环、同一准备结构”。

## 三、第三层提醒：共同扛板级问题的人，不要先按“异党”逻辑处理；很多嵌入式项目不是死于 bug，而是死于联合调试先分裂

原文说：
“共同抗日的党派就是友党，不是‘异党’。”

翻到嵌入式工程里，我觉得特别精准：
凡是共同扛系统主问题的人——
硬件、底层驱动、固件、应用、测试、运维、FAE、工厂支持、供应商接口——
都不该先被按“异己方”来处理。

很多嵌入式项目内耗最重的地方，就在这里：
- 硬件先把软件当背锅对象
- 软件先把硬件当甩不掉的坑源
- 测试先把研发当不听反馈的人
- 研发先把现场和工厂当只会制造噪音的一线
- 供应商信息一旦不透明，就先进入互相设防和争解释权

结果是什么？
系统真正的问题链还没打穿，
团队已经先把联合调试能力对消掉了。

这并不是说所有协作方都天然可信，
也不是说没有边界差异。
而是说，
如果一个嵌入式团队连谁是在共同扛主问题的人都不先分清，
它永远建立不起真正的联合调试能力。

## 四、第四层提醒：反对联调磨擦，不等于没有边界；嵌入式团队真正需要的是严格自卫，而不是软弱放任或报复式接口战争

原文对磨擦的处理非常完整：
反对互相对消力量的磨擦，
但若横逆一定要来，
就必须严正对待；
原则是“人不犯我，我不犯人；人若犯我，我必犯人”，
同时必须站在严格自卫立场，不能越界。

翻到嵌入式项目里，就是：
你当然不能把工程现场过成天天部门打架；
但也不能把“为了协作”理解成：
- 接口越权可以一直发生
- 无依据甩锅可以反复出现
- 现场事实可以被长期隐藏或改写
- 关键变更可以绕过配置和版本管理

真正成熟的工程边界感应该是：
- 不主动挑起接口战争
- 不把协作磨擦当主要工作方式
- 但若有人持续越界、持续破坏事实面、持续伤害主线排障
- 就必须有清楚的机制回应
- 回应目标是止损、恢复闭环、守住事实，不是报复性内斗

例如：
- 版本绕管控，要拉回配置管理
- 现场信息长期缺失，要拉回问题记录标准
- 反复甩锅无证据，要拉回复现和证据链规则
- 未授权改动破坏联调节奏，要让其承担明确成本

这就是嵌入式团队的严格自卫。
没有它，联合调试会被持续腐蚀。

## 五、第五层提醒：真正的统一，不是大家别吵，而是统一在主问题、联合调试和持续进步上

原文说：中国需要统一，
但应统一于抗战、统一于团结、统一于进步。

翻到嵌入式工程里，就是：
团队当然需要统一，
但不是统一于：
- 表面先别吵
- 谁都别提根因分歧
- 只要版本先能过就别再追深层结构问题
- 为了气氛稳定，先别动调试机制和协作规则

那种统一，
经常只是统一在假稳定，
统一在停滞，
统一在问题继续滚到下一版。

真正值得追求的统一，必须统一在三件事上：
- 当前主问题是不是被共同识别和守住
- 联合调试是不是在真实协同，而不是表面配合
- 系统是不是还在继续进步，而不是借口稳定停止升级

如果不是统一在这三件事上，
那就只是让整个团队一起把问题拖到以后更贵的时候爆。

## 六、如果把这篇文章压成今天嵌入式团队最好用的行动清单

我会把它压成六条工程提醒：

第一，别把“准备下一阶段”误成全面重构，先拆清现场、稳定性、架构、生产和组织几条战线。
第二，别把“靠自己扛”理解成继续透支少数高手，要把内部治理做成团队抵抗力。
第三，别把一起扛板级问题的人先当异己，先按友军逻辑组织联合调试。
第四，别让联调磨擦和甩锅对消主力；反过来，也别为了协作放弃接口和事实边界。
第五，所有边界回应都坚持严格自卫：够清楚、够止损、不过度扩大内耗。
第六，真正的统一只能统一在主问题、联合调试和持续进步上，而不是统一在表面不冲突上。

## 七、结语：真正成熟的嵌入式准备期，不是继续更拼命，而是把工程系统整到更能共同承压、共同调试、共同前进

读完这篇文章，我越来越强烈地感觉到，
很多嵌入式项目之所以在最危险的一段过去后还是没有真正稳住，
不是因为技术本身完全无解，
而是因为团队没有把准备期过成系统升级期。

它一边说要进入新阶段了，
一边还是靠少数人黑箱救火；
一边说大家要合作，
一边又把真正并肩排障的人先当异己；
一边说别再内耗，
一边关键边界、证据链和接口纪律又没有严格自卫规则。

这种工程系统，是积不出反攻能力的。

所以这篇文章对今天嵌入式团队最深的提醒，是：
真正成熟的准备，不只是更能熬夜，而是更会整工程系统。
你要把团队整成一个内部治理更清楚、联合调试更稳定、边界原则更可靠、还在持续进步的结构。
只有这样，暂时稳住才不会只是又一次侥幸止血，
而会真正变成下一阶段主动反攻的起点。
