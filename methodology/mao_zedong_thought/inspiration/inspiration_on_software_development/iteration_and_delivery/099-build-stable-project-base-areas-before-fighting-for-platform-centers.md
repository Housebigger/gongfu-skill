# 从《建立巩固的东北根据地》看软件项目：先建稳定项目根据地，再争夺平台中心和主干流量

日期：2026-05-15

原文对应：
《建立巩固的东北根据地》
原文链接：https://www.marxists.org/chinese/maozedong/marxist.org-chinese-mao-19451228.htm

写作背景：
这篇指示写在抗战胜利后东北局势急速复杂化的时点。文章没有把“东北重要”简单翻译成“马上占领大城市和交通干线”，而是判断现时条件下最可靠的路线，是在距离对方占领中心较远的城市和广大乡村建立军事政治根据地，并通过群众工作、调查研究、本地干部、军区划分、地方武装、生产和争取工人知识分子来站稳脚跟。

说明：
转译到软件开发，099提醒的是：项目进入新市场、新平台、新客户群或新组织阶段时，不要一开始就把全部资源压在主平台、主流量、最显眼的中心节点上。先建一个稳定项目根据地：有真实用户、有交付闭环、有本地化理解、有骨干队伍、有可持续供给，再去争取中心战场。

核心思想的当代转译：
如果把099翻成软件项目语言，我会概括成一句话：项目初入复杂战场时，先在可控场景里建稳定根据地，把用户、交付、团队、运维和供给做扎实，再逐步争夺中心流量和主干平台。

## 一、不要把“中心入口”误认为“项目根据地”

原文说，大城市和交通干线重要，但在现时条件下不是巩固根据地。软件项目里也常见类似误判：以为上了大平台、接入主渠道、拿到首页入口、进入集团主系统，就等于项目站稳了。其实中心入口往往规则重、干扰多、替代快，早期团队如果基础不稳，很容易被平台节奏拖垮。

项目根据地不是流量最大的地方，而是你能持续交付、持续学习、持续留住用户的地方。

## 二、先找远离主冲突的可控场景，完成初步可靠创建

原文要求在一年内完成初步可靠的创建工作。软件项目也需要这种“初步可靠”：一个小但真实的用户群，一个可重复交付的版本节奏，一套能定位问题的日志和反馈机制，一条清楚的支持链路。它未必 glamorous，但能让团队真正立住。

没有可控根据地就去打中心战，常常会变成把所有 bug 和组织弱点暴露在最大压力下。

## 三、项目重心要转到用户工作，而不是只盯技术占点

原文第三条说，在力量已有发展后，东北工作重心是群众工作。软件项目对应的就是：当最初版本、人员和部署有了雏形之后，重心不能继续停留在“我们已经进场”上，而要转到真实用户工作——理解流程、解决痛点、给用户看得见的收益、组织反馈、沉淀用例。

用户没有获得看得见的利益，就不会因为你技术正确而天然拥护项目。

## 四、外来团队必须调查本地流程，不能拿旧项目经验压新场景

原文批评初到东北的干部地理民情不熟，又对艰苦工作不耐心。软件团队进入新业务、新行业或新客户现场时，也很容易带着旧框架套新问题：不了解审批链、数据口径、现场例外、真实使用人，却急着让对方适配系统。

新项目最怕的不是代码写得慢，而是团队不肯承认自己还不懂现场。

## 五、组织上要区分野战军和地方军，让团队有确定区域和任务

原文第五条提出划分军区、军分区，把部队分为野战军和地方军，并让一切军队有确定地区和任务。软件团队也要区分核心攻坚队、场景交付队、运维支持队、客户成功和本地实施，而不是所有人一起追最热问题。

任务区域一清楚，团队才可能和真实用户结合，项目才会从一次性攻坚变成持续运营。

## 六、交付人员不能只消耗业务供给，也要建设生产能力

原文第六条提醒大量脱产人员专靠地方供给不可持久，必须从事生产。软件项目里，如果团队只是不断消耗业务方时间、客户耐心、运维资源，却不能形成文档、工具、自动化、模板、培训和可复用模块，就会越做越贵，越做越沉。

稳定项目根据地必须能自己生产一部分供给，而不是永远等别人喂资源。

## 七、别放弃中心里的关键人才和知识力量

原文最后说，对大城市和交通干线的工人、知识分子仍要充分注意。软件项目即使先在边缘场景扎根，也不能和平台架构师、核心业务专家、安全合规、数据治理、关键用户代表断线。这些人决定未来项目能不能从根据地走向中心。

099给软件团队的提醒是：先建根据地不是逃避中心战，而是为了用更稳的用户基础、交付纪律、本地理解和人才连接，准备以后真正打得起中心战。
