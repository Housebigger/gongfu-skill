# 从《在不同地区实施土地法的不同策略》看软件项目推进：不要把同一套硬办法同时压到所有项目区域；成熟模块、半熟链路和新开局，要分类施策、分阶段推进

日期：2026-05-19

原文对应：
《在不同地区实施土地法的不同策略》
原文链接：https://www.marxists.org/chinese/maozedong/marxist.org-chinese-mao-19480203.htm

写作背景：
1948年2月3日，毛泽东在给刘少奇的电报里谈《在不同地区实施土地法的不同策略》。这篇文章最硬的地方，不只是强调“要推进土地法”，而是强调：同一项大任务，不能不看地区条件、群众基础、既有进度和主要矛盾，就拿同一套办法到处往下压。毛泽东把当时的地区分成三类：老解放区，大体已分过土地，重点不在重新大拆大翻，而在调整局部、整顿组织、解决党群矛盾；半老区，群众已发动、土地问题初步解决但不彻底，可以比较完整地实行土地法，并准备复查和再分；新解放区，则群众尚未发动、旧势力仍强，一开始不能企图“一下实行土地法”，而要分阶段推进，先立基础、先打主要对象、先组织骨干，再逐步深入。文章最后还特别强调，不论老区、半老区还是新区，太急了，都办不好。

说明：
把这篇文章转译到今天的软件项目推进语境里，当然不能机械照搬其中关于土地改革和战争年代农村工作的历史场景。但它背后那套方法，对今天做平台迁移、系统重构、流程整改、组织换轨、版本治理和复杂交付的人，非常有现实感。很多软件项目真正做坏，不是因为团队不肯推进，而是因为负责人只会一种硬办法：不分成熟模块、半熟链路和新开局，不分哪些地方该调旧账、哪些地方适合深推、哪些地方必须先立基础，就把统一重构、统一治理、统一流程、统一节奏同时压下去，最后把本来能做成的项目也做翻了。

核心思想的当代转译：
如果把《在不同地区实施土地法的不同策略》翻成今天软件项目的语言，我会把它概括成一句话：真正成熟的项目推进，不是拿同一套硬治理、硬重构、硬流程同时压到所有模块、团队和阶段，而是先分清哪些区域已经成熟、哪些只是半熟、哪些还刚开局，再决定哪里该补旧账、哪里该深推进、哪里该分阶段立基础；越是一刀切、越想一把梭，越容易把原本能推进的局面推进成系统性事故。

## 一、第一层提醒：不是所有项目区域都处在同一成熟度，别拿一套总方案横扫全栈

117 一上来先分类地区，这个动作翻到软件里特别值钱。

真正成熟的项目负责人，
第一步不是先宣布“大整顿马上开始”，
而是先问：
眼前这些地方，到底是不是同一种性质、同一个阶段、同一种可动程度？

软件项目里最常见的误判是：
- 一个核心模块老了，就想全系统同步重构
- 一条交付链上有几个关键瓶颈，就想研发、测试、运维、产品、支持全部统一重排
- 一个新平台刚跑出点苗头，就想顺手把权限、流程、接口、数据模型、组织角色一起彻底换掉
- 一轮事故暴露出问题，就想所有团队一夜之间统一改制度、统一补规范、统一改代码风格

看上去很有决心，
其实常常只是没有分类。

项目真相往往是：
- 有些模块已经跑了很多年，问题是旧债和接口脏活
- 有些链路已经改出一半，适合继续深推
- 有些新项目刚启动，连真实边界和主问题都还没摸清

不分类，
就等于把成熟区、过渡区和新开局混成一锅处理。

## 二、第二层提醒：对已经跑过一大段的成熟区，重点往往不是推倒重来，而是调旧账、整接口、修组织协作

毛泽东谈老解放区时说得很清楚：
大体早已分过土地，
重点不在重新大拆大翻，
而在调整局部、整顿队伍、解决关系矛盾。

翻到软件里，
这对应的是那些已经长期在线、已经承载真实业务、已经形成真实依赖的成熟区。

这类区域最怕的误判是：
“既然还有问题，那说明过去全白干了，不如整体重写。”

但成熟区真正需要的，
很多时候不是总重构，
而是：
- 把关键旧坑补掉
- 把最危险的耦合拆开
- 把接口边界重新标清
- 把文档、值班、发布、回滚这些旧账补齐
- 把角色分工和协作责任重新理顺

现实里常见的成熟区例子包括：
- 已上线多年的核心交易链路
- 被很多团队依赖的内部平台
- 已经被客户深度接入的老 API
- 看似稳定但长期靠少数人硬扛的历史系统

这些地方最危险的错误，
往往不是动得太少，
而是把“应当局部纠偏”做成“全盘推翻重来”。

## 三、第三层提醒：对已经发动起来但还没彻底改透的半熟局，要敢深推，但也要准备复查、返工和二次分配

117 谈半老区时非常务实：
可以比较完整地实行土地法，
但也要准备复查一两次、再分一次。

翻到软件项目里，
半熟局就是那些已经改出基础、但还没真正改透的区域。

比如：
- 服务拆分已经开始，但 ownership 还没压实
- 测试体系已经补了一轮，但关键回归链路还不稳定
- 发布流程已经部分规范，但事故复盘、值班权责、回滚条件还没打通
- 平台化已经推进，但老系统迁移、数据清洗、监控口径仍旧半拉子

这种局面最需要的不是保守，
而是继续推进；
但推进时又不能幻想“一次上线定终局”。

真正成熟的做法是：
- 允许第一轮拆分还不够干净
- 允许第一轮治理会漏问题
- 允许第一次流程落地后还要复盘返工
- 把二次迁移、二次梳理、二次收口，视作正常过程

很多项目不是败在不肯改，
而是败在太想一次改成。

半熟局的硬规则不是“别再动”，
而是“继续深推，但把复查机制一开始就准备好”。

## 四、第四层提醒：对刚打开的新局，先立基础、先抓主问题、先组织骨干，不要一上来就全面制度化和平台化

117 最值钱的一刀，
就在新区策略：
不能企图一下实行土地法，
要分阶段推进，
先立基础、先分对象、再逐步深入。

翻到软件项目里，
这几乎可以直接当成新开局治理规则。

一个新项目、新产品线、新平台、新架构尝试，
最先要做的往往不是“把完整秩序一次做完”，
而是“让这个局先能跑起来”。

这意味着：
- 新产品刚起步，先抓核心用户路径，不要第一周就做全套平台治理
- 新平台刚试行，先把主链路监控、回滚和交付责任打实，不要先统一所有外围系统
- 新组织刚接手项目，先搞清最关键的失序点和主要破坏环节，不要对所有人同步下最重改造
- 新流程刚启动，先证明它能支撑关键发布和关键恢复，不要先追求全公司统一模板化

新开局最容易犯的错就是：
基础还没立住，
就急着全面铺开；
骨干还没组织出来，
就急着所有战线一起变；
主问题还没抓住，
就急着把所有旧账同时清完。

这样做，
往往不是快，
而是把所有阻力一起激活。

## 五、第五层提醒：不同成熟区里的可用力量也不一样，别把老系统维护者、中坚工程师和可争取协作者一锅推开

117 不只讲阶段，
还讲不同地区里不同力量该怎样分别对待。

翻到软件项目里，
这层提醒非常现实：
不同区域里，
你可用的人、该争取的人、该重点处理的问题，
都不一样。

很多项目一进入整改，
最容易犯的粗糙错误就是：
- 把所有维护老系统的人都当成阻力
- 把所有对新方案有疑问的人都归成保守派
- 把所有中间力量都看成不坚定、没价值
- 把所有外部依赖方都视为必须立刻清理的包袱

可现实里真正成熟的做法是：
- 有些老维护者掌握关键脏知识，应该组织进旧账清理
- 有些中间工程师虽然不激进，但能成为稳定推进的主干
- 有些产品、测试、运维或客户成功角色，是新旧过渡期必不可少的缓冲力量
- 真正该重点打击的，往往是持续制造混乱、拒绝边界、破坏主线的人和机制，而不是所有“旧人旧系统”本身

不会区分力量的团队，
最后常常不是把项目做纯，
而是把支持面做没。

## 六、第六层提醒：太急了，往往不是更有执行力，而是更容易把项目推成事故

117 最硬的一句收尾是：
太急了，必办不好。

这句话翻到软件里，
几乎就是很多失败项目的复盘标题。

很多系统性事故并不是方向错了，
而是节奏错了：
- 还没分区，就统一重构
- 还没站稳，就全量切换
- 还没形成新交付秩序，就先把旧秩序全部清零
- 还没组织出第二层骨干，就把所有责任都往新系统上压

急，
经常伪装成魄力；
但很多时候，
它只是没有耐心把事情按阶段做完。

真正成熟的软件推进，
不是不敢碰大问题，
而是知道：
项目越复杂、依赖越重、历史越深，
越不能靠同一套硬办法一把梭到底。

## 结语：软件推进不是统一猛推，而是分清成熟模块、半熟链路和新开局，再决定不同节奏

所以，《在不同地区实施土地法的不同策略》给今天软件团队的硬提醒是：不要把同一套硬重构、硬治理、硬流程同时压到所有项目区域。已经成熟的地方，重点往往是调旧账、整接口、修协作；已经半熟的地方，适合继续深推，但要准备复查、返工和二次收口；刚打开的新局，必须先立基础、先抓主问题、先组织骨干，再逐步深入。真正成熟的项目推进感，不是一刀切的狠，而是分得清区域成熟度、拿得准推进次序、压得住统一猛推的冲动。
