# 从《关于健全党委制》看产品迭代中的技术决策：建立团队集体决策机制，防止个人独断包办

日期：2026-06-12

原文对应：
《关于健全党委制》
原文链接：https://www.marxists.org/chinese/maozedong/marxist.org-chinese-mao-19480920.htm

写作背景：
1948年9月20日，毛泽东为中共中央起草了《关于健全党委制》的决定。这篇文章很短，但组织方法极硬。它不是泛泛地说"要民主"，而是极其尖锐地指出了一个普遍存在的组织病：很多党组织长期被个人包办，重要决定不经集体讨论，书记一个人说了算。毛泽东说得很直白："党委制是保证集体领导、防止个人包办的党的重要制度。有些（当然不是全部）领导机关，个人包办和个人解决重要问题的习气甚为浓厚，重要问题的解决，不是由党委会议做决定，而是由个人做决定，党委委员等于虚设。"他提出的具体办法是：各级党委必须建立定期的党委会议制度，一切重要问题必须经党委集体讨论和决定，不能由个人说了算。会议次数要够，议题要提前通知，委员要有充分准备，不能搞突然袭击。这不是否定个人负责制，而是在集体决定之后，再由个人分工去执行。

说明：
把这篇文章转译到软件产品迭代的技术决策里，现实感极强。很多团队的技术方向、架构选择、版本规划、资源分配，长期由一个人——通常是首席架构师或技术负责人——独自拍板，其他工程师只是被动执行。130提醒软件团队：技术决策的质量，不取决于那个拍板的人有多聪明，而取决于决策过程是否经过了集体讨论、多方信息输入和充分质疑。一个架构师再强，他的信息也是有限的、视角也是单一的、盲区也是必然的。把重大技术决策交给一个人独断，和把产品成败押在一个人的认知上限上，本质上是一回事。

核心思想的当代转译：
如果把《关于健全党委制》翻成今天软件团队技术决策的语言，我会把它概括成一句话：所有重要的技术决策——架构选型、版本规划、技术债务处理策略、关键模块的取舍——必须经过团队集体讨论后决定，不能由任何一个人独断包办；集体讨论不等于人人拍板，而是在充分讨论后形成共识或明确分歧，再由负责人拍板执行。

## 一、第一层提醒：130 不是否定技术负责人，而是防止"一个人说了算"的决策病

130 最硬的一刀，不是取消个人负责，而是把"个人包办"定性为一种组织病。

毛泽东说得很清楚：不是不要书记，不是不要个人分工负责，而是"一切重要问题均须交委员会讨论，由到会委员充分发表意见，做出明确决定，然后分别执行"。

翻到软件团队里，这句话极其值钱。很多团队的技术决策模式，表面上有评审会、有技术方案讨论，实际上：

- 架构师写完方案直接推进，评审会只是走个过场，大家点个头就算通过；
- 技术负责人在脑子里已经决定了技术选型，开会只是为了"通知"团队，不是真的在听取不同意见；
- 版本规划由产品负责人一个人拍板，工程师只负责"接单"，对为什么做这个不做那个没有发言权；
- 更常见的是：重大技术决策根本没有正式的讨论环节，都是在走廊里、微信群里、或者一个人的脑子里完成的。

130提醒我们：
技术负责人的价值，不是替团队做所有决定，
而是确保所有重要决定都经过了充分的集体讨论。
个人的认知上限，不应该成为团队决策质量的天花板。

## 二、第二层提醒：什么是"重要问题"——不是每个工单都要开会讨论

毛泽东在文章里说的是"一切重要问题"交委员会讨论。他没有说每一件小事都要集体决策。

翻到软件团队，这层意思非常关键。很多团队一听到"集体决策"，本能反应就是"效率要下降了"、"每个需求都要开会讨论"。但 130 的逻辑恰好相反：

需要集体讨论的，是那些影响全局的重要决策：
- 架构选型：用什么技术栈、是自研还是用开源方案、是微服务还是单体——这些决定了团队未来几个月甚至几年的工作方式；
- 版本规划：这个版本的核心目标是什么、决定性功能是哪个、哪些功能可以延后——这些决定了团队接下来几周的时间分配；
- 技术债务处理策略：什么时候还债、还多少、哪些债必须先还——这些决定了系统的长期健康度；
- 关键模块的取舍：某个功能是做还是砍、某个系统是重构还是打补丁——这些决定了产品的走向。

不需要集体讨论的，是执行层面的具体决策：
- 一个 bug 用方案 A 还是方案 B 修复——由负责的工程师决定；
- 一个功能的代码组织方式——由做 code review 的人决定；
- 一天的任务优先级——由工程师自己安排。

130提醒我们：
集体讨论的是方向性、全局性的重要问题，
不是每一个具体执行细节。
分清"什么需要讨论"和"什么不需要讨论"，
本身就是一种决策能力。

## 三、第三层提醒：讨论不是走过场，必须让与会者提前拿到材料、有充分准备时间

毛泽东在文章里专门提了一个细节："委员间要有充分的会前酝酿和讨论，不要临时提议，仓促通过。"

这句话翻到软件团队，对应的是一个极其普遍的问题：评审会变成了"现场第一次看到方案"的会。

现实中的场景：

- 架构评审会上，架构师现场展示方案，其他人是第一次看到细节，根本来不及思考和质疑；
- 版本规划会上，产品经理现场抛出需求列表，工程师没有提前看过，只能在会议上凭直觉给出排期；
- 技术方案评审变成了"谁说话声音大谁赢"，因为没有提前准备，讨论变成了辩论赛而不是信息对齐；
- 更常见的是：关键决策根本没有文档，全在决策者的脑子里，其他人想提意见都找不到抓手。

130提醒我们：
有意义的集体讨论，前提是与会者提前拿到材料、有时间思考。
没有准备的讨论，不是集体决策，是集体表演。
真正值钱的意见，是在看到方案后花时间想出来的，
不是在会议上凭第一反应说出来的。

## 四、第四层提醒：集体讨论之后，必须有明确的决定，不能议而不决

毛泽东在文章里反复强调的是"做出明确决定，然后分别执行"。不是讨论了就算了，而是讨论之后必须形成结论。

翻到软件团队，这对应的是另一种常见病：讨论了很多，但没有结论。

- 技术评审会开了两个小时，各方观点都表达了，但没有形成最终决定，方案还是悬着；
- 版本规划会上争论了半天优先级，但谁都不拍板，最后还是回到"原来怎么排就怎么排"；
- 架构选型讨论了好几轮，正反两方都有道理，但没有人做最终裁决，项目在这件事上停了三周；
- 更常见的是：会议上说了"回去再研究研究"，但没有人负责跟进，下一次开会还是讨论同样的问题。

130提醒我们：
集体讨论的目的，不是为了"每个人都发表意见"，
而是为了"在充分讨论后做出明确决定"。
讨论是手段，决定才是目的。
如果讨论了半天没有结论，那比不讨论还糟——
因为大家花掉了时间，却没有推进任何事。

## 五、第五层提醒：决定之后，分工执行——集体决策不等于集体不负责

130 的后半段逻辑是：集体讨论做决定，然后个人分工执行。

毛泽东说得很清楚：委员会做出决定之后，"由到会委员充分发表意见，做出明确决定，然后分别执行"。每个人在自己的分工范围内负责落实，不是集体讨论完了就集体散了。

翻到软件团队，这层意思解决的是"集体决策导致无人负责"的常见困境：

- 技术方案是大家一起讨论的，但出了问题没有人觉得自己应该负责；
- 版本目标是团队一起定的，但延期的时候每个人都能找到理由说"这不是我的问题"；
- 架构选型是集体投票的，但线上出事故的时候，没有人觉得"这票我投错了"；
- 更常见的是：集体决策变成了一种"责任稀释"机制——大家一起决定，等于没有人单独负责。

130提醒我们：
集体决策 + 个人分工执行 = 正确的模式。
讨论的时候大家一起，执行的时候各管一段。
谁执行哪一块，谁就对那块的结果负责。
集体讨论不稀释责任，而是让责任分配得更清楚。

## 六、第六层提醒：为什么"个人包办"的诱惑力如此之大——效率幻觉

130 最难执行的部分，不是"要不要集体讨论"，而是"在时间紧迫的时候，还能不能坚持集体讨论"。

毛泽东写这篇文章的时候，正是解放战争最紧张的阶段——辽沈战役刚开打，淮海战役即将开始，平津战役迫在眉睫。越是这种时候，"个人拍板效率更高"的诱惑力越大。

翻到软件团队，这个场景每天都在发生：

- 版本快上线了，出了一个架构问题，技术负责人直接拍板改了方案，没有时间讨论；
- 客户催着要功能，产品经理一个人决定了需求范围，没有和工程师商量；
- 线上出了事故，架构师独自从头到尾处理完了，其他人只是旁观；
- 更常见的是：平时不建立集体讨论的机制，遇到紧急情况自然就回到"谁最熟谁拍板"的模式。

130提醒我们：
越是时间紧、压力大的时刻，越不能放弃集体讨论。
不是说要慢慢开会讨论半天，
而是哪怕只花十五分钟拉上两三个人快速对齐，
也比一个人独断要好得多。
因为紧急情况下的个人独断，往往是在信息不完整、压力过大的状态下做出的，
错误概率比平时更高。

## 结语：130 给软件团队最硬的提醒，是建立制度化的集体决策机制，而不是依赖个人的英明判断

所以，130 对今天软件团队最现实的提醒是：所有重要的技术决策——架构选型、版本规划、技术债务策略、关键模块取舍——必须经过团队集体讨论后决定，不能由任何一个人独断包办。讨论之前，材料要提前发给与会者，让大家有时间准备；讨论之中，每个人要充分发表意见，不能走过场；讨论之后，必须形成明确决定，不能议而不决；决定之后，各人分工执行，集体讨论不稀释个人责任。越是时间紧、压力大的时刻，越不能放弃这个机制。因为团队的决策质量，不取决于那个最聪明的人的上限，而取决于决策过程是否让足够多的信息、视角和质疑进入了讨论。一个架构师再强，他的盲区也是必然的。制度化的集体决策，就是用团队的集体智慧去覆盖个人的必然盲区。
