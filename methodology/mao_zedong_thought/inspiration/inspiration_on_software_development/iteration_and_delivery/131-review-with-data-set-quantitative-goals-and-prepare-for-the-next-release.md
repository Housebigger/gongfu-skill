# 从《中共中央关于九月会议的通知》看产品团队阶段复盘：用数据复盘、设定量化目标、为下个版本做好准备

日期：2026-06-12

原文对应：
《中共中央关于九月会议的通知》
原文链接：https://www.marxists.org/chinese/maozedong/marxist.org-chinese-mao-19481010.htm

写作背景：
1948年10月10日，毛泽东为中共中央起草了《中共中央关于九月会议的通知》，向全党通报1948年9月中央政治局会议（即"九月会议"）的主要决议。这篇通知不是一份泛泛的政治宣言，而是一份极其硬核的阶段复盘加下一阶段动员文件。毛泽东在通知中做了三件事：第一，用具体数据总结过去两年的战果——歼敌多少个旅、解放多少座城市、土地改革推进到什么程度、军队扩充到多少万人，全部用事实说话，没有一句空洞的"形势大好"。第二，根据这些事实，对当前阶段做出明确判断——解放战争已经到了"从根本上打倒国民党反动派"的转折点，但距离最终胜利还需要一定时间，不能因为形势好就松懈，也不能因为还要打就悲观。第三，给下一阶段设定了具体的量化目标和行动要求——再歼敌多少个师、再解放多少个地区、军队要怎么整编、财政经济要怎么统一、干部要怎么准备。整篇通知的结构就是：用数据复盘上一阶段 → 基于事实判断当前阶段 → 设定量化目标并部署下一阶段。这个结构，今天看起来仍然是一份标准的"阶段复盘 + 下一阶段OKR"模板。

说明：
把这篇文章转译到软件产品团队的阶段复盘中，现实感极强。很多产品团队做版本复盘、季度复盘或阶段总结时，最常见的问题不是不复盘，而是复盘没有数据、目标没有量化、下一阶段的准备没有落地的具体安排。复盘变成了"大家辛苦了"的鼓劲会，目标变成了"争取做得更好"的口号，下一阶段的准备变成了"我们接着干"的表态。131提醒产品团队：真正有价值的阶段复盘，必须用数据说话——上一个版本交付了多少、用户反馈怎么样、核心指标变化如何；基于这些数据判断当前阶段的位置——是离目标更近了还是更远了，决定成败的结构性问题是什么；然后为下一阶段设定明确的量化目标——下个版本的核心指标要达到什么数值、关键功能要交付到什么程度、团队要怎么调整。没有数据的复盘是空谈，没有量化目标的部署是空转。

核心思想的当代转译：
如果把《中共中央关于九月会议的通知》翻成今天产品团队阶段复盘的语言，我会把它概括成一句话：阶段复盘必须用具体数据回顾上一阶段的交付结果，基于事实判断当前进度和结构性问题，然后为下一阶段设定明确的量化目标和资源部署——不用数据的复盘是空谈，不设量化目标的部署是空转。

## 一、第一层提醒：复盘的第一步不是谈感受，而是把数据摆出来

毛泽东在通知里做的第一件事，不是分析形势、不是鼓舞士气，而是摆数据。歼敌数字、解放城市数量、军队规模、土改进度、财政状况——全部用具体数字说话。

翻到产品团队的阶段复盘里，这句话极其值钱。因为绝大多数复盘会的开场是这样的：

- "这个版本大家都很拼，加班很多，辛苦了。"
- "整体来说，用户反馈还不错。"
- "上线还算顺利，没有出什么大问题。"
- "我们完成了大部分需求，还有几个延到了下个版本。"

这些话的共同特点是什么？没有数据。

真正有价值的复盘开场应该是这样的：

- 这个版本交付了12个需求，其中8个按期上线，4个延期，延期原因分别是……
- 上线后首周日活从50万涨到53万，涨幅6%，核心转化率从3.2%升到3.5%；
- 用户投诉工单从上线前日均40单上升到日均65单，其中新增投诉集中在搜索模块；
- 线上故障2次，P1级别1次，影响时长47分钟，根因是缓存击穿；
- 技术债务本版本削减了3项，新增了2项，净减少1项。

131提醒我们：
复盘的第一步不是谈感受，而是把数据摆出来。
没有数据的复盘，本质上是在用印象代替事实。
"感觉还不错"和"数据证明还不错"，是完全不同的两件事。
数据不需要完美，但必须要有。
粗略的数据也比精准的印象可靠。

## 二、第二层提醒：数据不是用来粉饰太平的，而是用来暴露结构性问题的

毛泽东在通知里不只报喜，也报了实际问题：解放区财政困难、干部数量不足、某些地区的土改执行有偏差。这些不是用来"表态承认困难"的装饰，而是用来识别下一阶段要解决的结构性矛盾。

翻到产品团队的复盘中，这层意思对应的是一个极其普遍的问题：很多团队的数据复盘，变成了"挑好看的数据讲"：

- 日活涨了，大讲特讲；留存降了，一笔带过；
- 新功能上线了，大讲特讲；bug数量上升了，归因到"测试资源不足"就过去了；
- 大客户反馈好了，大讲特讲；中小用户流失了，没有数据所以也不提；
- 交付速度快了，大讲特讲；代码质量下降、技术债积累，不在复盘会上讨论。

这种"选择性呈现数据"的复盘，和不做复盘差不多——因为真正决定项目下一阶段走向的，往往不是那些好看的数据，而是那些被跳过的问题。

131提醒我们：
数据的价值不在于证明"我们做得不错"，
而在于暴露"我们在哪里还有结构性问题"。
日活涨了但留存降了，说明新增用户没接住；
交付快了但质量降了，说明团队在吃老本；
大客户满意但中小用户流失，说明产品在脱离主流场景。
复盘数据的时候，最值得关注的数据，
恰恰是那些"不好看"的数据。
因为那些数据背后，藏着下一阶段最需要解决的问题。

## 三、第三层提醒：基于数据判断当前阶段位置——是离目标更近了，还是只是更忙了

毛泽东在通知里做了一件非常关键的事：基于战果数据，判断当前解放战争处于什么阶段。他的结论是：已经到了从根本上打倒国民党的转折点，但还没有到最终胜利，接下来还要继续打硬仗。

这个判断的厉害之处在于：它既不是盲目乐观（"我们快赢了"），也不是保守悲观（"还要打好久"），而是基于事实的阶段定位。

翻到产品团队的复盘中，这层意思解决的是一个常见问题：团队分不清"更忙了"和"更接近目标了"。

很多团队复盘时的表述是：

- "我们这个版本做了很多事。"——做了很多事，但是不是做了对的事？
- "团队加班很多，很辛苦。"——辛苦不等于有进展，忙不等于在前进；
- "需求基本都交付了。"——交付了需求，但核心指标有没有变化？
- "用户反馈比上个版本好。"——好了多少？好在哪？核心痛点有没有被解决？

131提醒我们：
阶段复盘的核心判断，不是"我们做了多少事"，
而是"我们离目标更近了多少"。
做了20个需求但核心指标没变，说明这20个需求可能不是决定性的；
加班很多但交付质量下降，说明团队在用透支换产出；
看起来很忙但关键路径上的问题没解决，说明忙在了错误的方向上。
真正有价值的阶段判断只有一个标准：
这个阶段结束后，产品距离目标是不是真的更近了。
如果答案是"是的"，数据会证明；
如果答案是"不确定"，那说明上一阶段的目标和衡量标准本身就有问题。

## 四、第四层提醒：下一阶段必须设定量化目标，不能只有方向没有刻度

毛泽东在通知里最硬的部分，是他给下一阶段设定了极其具体的量化目标：再歼敌多少个旅、再解放多少个省、军队要扩充到多少人、干部要准备多少名。每一个目标都有数字，不是"继续努力"，不是"加大力度"。

翻到产品团队的复盘中，这对应的是几乎所有团队都犯过的错：复盘完了，下一阶段的目标写成：

- "提升用户体验。"——提升到什么程度？用什么指标衡量？
- "优化系统性能。"——优化到什么数值？延迟降多少？吞吐量提多少？
- "加强用户增长。"——增长多少？从多少到多少？什么渠道？
- "提高代码质量。"——怎么衡量？bug率降到多少？测试覆盖率到多少？

这些没有数字的目标，和没有目标是一样的。因为没有人知道"做到了"长什么样，也没有人知道"没做到"该追谁的责。

131提醒我们：
下一阶段的目标，必须是量化的。
"提升用户体验"不是目标，"用户满意度从3.8分升到4.2分"才是；
"优化性能"不是目标，"首页加载时间从2.1秒降到1.2秒以内"才是；
"加强增长"不是目标，"月活从100万增长到130万"才是。
量化目标的本质，不是给团队加压力，
而是让所有人知道"做成了"的标准是什么。
没有刻度的方向，不是方向，是口号。
有刻度的方向，才能变成团队共同的行动标尺。

## 五、第五层提醒：量化目标不只是数字，更要拆到具体的行动部署和责任人

毛泽东在通知里不只是说"再歼敌一百个旅"就完了，他还部署了具体的行动：各个野战军分别负责哪个方向、军队怎么整编、财政经济怎么统一、干部怎么培训和分配、新区工作怎么开展。每一个量化目标后面，都跟着具体的行动安排和责任人。

翻到产品团队，这层意思对应的是：很多团队定了量化目标之后，就以为工作做完了。目标写在PPT里、贴在墙上、发在群里，但没有拆解成具体的行动和责任人。

现实中的场景：

- 目标是"月活增长30%"，但没有拆到具体是哪个功能驱动增长、哪个渠道拉新、哪个运营动作促活；
- 目标是"crash率降到0.1%以下"，但没有指定哪些模块要重构、哪些依赖要替换、谁负责排查top crash；
- 目标是"用户满意度提升0.5分"，但没有说哪些用户反馈要优先解决、哪些流程要优化、谁负责跟进；
- 更常见的是：量化目标定完了，但资源分配没有变化——人还是那些人、排期还是那个排期、优先级还是那个优先级，目标变成了挂在墙上的数字。

131提醒我们：
量化目标只是起点，不是终点。
目标定了之后，必须回答三个问题：
第一，这个目标要通过哪些具体行动来实现？
第二，每个行动由谁负责、什么时间完成？
第三，现有的资源分配需不需要调整来支撑这些行动？
如果目标定了但行动没变、资源没动、人不调整，
那这个目标就不是用来执行的，而是用来汇报的。
真正有效的阶段部署，是目标 + 行动 + 责任人 + 资源调整四位一体。

## 六、第六层提醒：阶段复盘不只是向后看，更是为下一阶段做好准备——团队状态、能力储备和组织调整

毛泽东在通知里有一大段内容不是谈军事战果，而是谈组织准备：干部要准备多少、怎么训练、怎么分配；军队要怎么整编、怎么提升正规化水平；新解放区的接收和管理要怎么准备。他非常清楚：下一阶段的仗能不能打赢，不只取决于上一阶段的战果，还取决于组织有没有为下一阶段做好准备。

翻到产品团队，这层意思是最容易被忽视的：

大多数团队的阶段复盘，只关注"上个版本做了什么"和"下个版本做什么"。但很少有人问：

- 团队的整体状态怎么样？连续高强度交付后，团队是精力充沛还是已经透支？
- 团队的能力储备够不够？下一阶段如果有新的技术挑战或新的业务方向，现有的技能能不能支撑？
- 组织结构需不需要调整？上一阶段的团队配置是为上一阶段的问题设计的，下一阶段的问题可能需要不同的配置。
- 工具链和基础设施需不需要升级？上一阶段暴露出的效率瓶颈——构建速度慢、测试环境不稳定、发布流程繁琐——下一阶段能不能解决？
- 知识和经验有没有沉淀？上一个版本踩过的坑、积累的经验，有没有被记录下来，还是只存在几个人的脑子里？

131提醒我们：
阶段复盘的价值，不只是总结过去，
更是为下一阶段做好准备。
产品要准备，团队也要准备。
技术要准备，组织也要准备。
上一阶段的胜利，不自动带来下一阶段的胜利。
如果团队状态透支了、能力没跟上、组织没调整、工具没升级，
那下一阶段开始的时候，上一阶段积累的优势就会被消耗掉。
真正成熟的团队，在复盘的时候会花同样的精力
回答两个问题：上一阶段做对了什么？下一阶段我们准备好了吗？

## 结语：131给产品团队最硬的提醒，是用数据复盘、设定量化目标、为下一阶段做好全面准备

所以，131对今天产品团队最现实的提醒是：阶段复盘不能只谈感受，必须用数据说话。数据不是为了粉饰太平，而是为了暴露结构性问题。基于数据判断当前阶段的位置——是离目标更近了，还是只是更忙了。下一阶段必须设定量化目标，不能只有方向没有刻度。量化目标要拆到具体行动、责任人和资源调整，不能只挂在墙上。阶段复盘不只是向后看，更是为下一阶段做好全面准备——团队状态、能力储备、组织调整、工具链升级，每一项都要检查。不用数据的复盘是空谈，不设量化目标的部署是空转，不做准备就开始的下一阶段是用上一阶段的积累为下一阶段的失误买单。真正有效的阶段复盘，是用事实校验信心，用数据识别问题，用量化目标锁定方向，用行动部署和资源调整确保执行，最后用团队和组织的全面准备为下一阶段的胜利打底。
