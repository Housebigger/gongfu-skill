# 从"围而歼之"看软件开发：包围技术难题再进攻，堵死回退路径，协调多团队

- 日期：2026-06-12
- 原文对应：《关于辽沈战役的作战方针》（1948 年 9 月、10 月，毛泽东给林彪、罗荣桓的电报）及《关于淮海战役的作战方针》（1948 年 10 月 11 日）
- 原文链接：https://www.marxists.org/chinese/maozedong/marxist.org-chinese-mao-194809_10.htm
- 写作背景：毛泽东在辽沈战役和淮海战役中反复强调"先围后打"——先切断敌人退路和补给线，形成合围，再发起总攻。打锦州时要求"先打锦州、榆关、唐山"，切断东北与关内的联系；淮海战役要求各纵队协同，"不要让敌人跑掉"。
- 说明：本文把毛泽东"包围歼灭"的军事方法论，转译为技术团队攻克难题、堵死回退路径、协调跨团队协作的实战指导。
- 核心思想当代转译：不要正面硬冲技术难题——先从侧面和后方建立控制点，切断问题的回退路径，让团队形成合围后再发起总攻。同样，跨团队协作不是开个会就完事，而是让每个团队像军事上的各纵队一样，明确自己的包围面和进攻时机。

---

## 一、不要正面硬冲——先包围再进攻

毛泽东打锦州，不是从沈阳一路正面推进，而是先切断北宁线，让锦州变成一座孤城——然后再打。正面推进看似直接，实则每前进一步都要分兵防守侧翼和补给线，越打越散。先切断外围联系，守军就成了瓮中之鳖。

- **正面硬冲的表现**：拿到一个复杂技术难题，直接上手写代码，边写边踩坑，踩了一个又一个，最后发现自己在一个越来越深的洞里。每修一个 bug 引出两个新 bug，每加一层抽象发现底层还要改。进度条不动，代码越来越乱，时间一天天过去
- **包围再进攻的表现**：
  - 先用两天做 spike，画出问题的依赖关系图——哪些模块涉及、哪些接口要改、哪些数据要迁移。依赖图就是战场地图，没有地图就冲锋是送死
  - 先写集成测试把现有行为锁住——这样改动一旦破坏已有功能，测试立刻报警。测试就是包围时的火力封锁线，封锁线没拉好就冲锋，敌人从侧翼穿插你都不知道
  - 先建立 feature flag 和灰度发布机制——新方案上线后可以随时切回旧方案。灰度就是包围时的预备队，冲锋受阻时可以撤回重组
  - 先做技术方案的"纸上推演"——把所有可能的失败模式列出来，逐一确认应对方案。军事上叫兵棋推演，软件上叫 failure mode analysis
  - 先梳理外部依赖——哪些第三方服务会受影响、哪些团队需要提前通知。军事上叫侦察敌情，不了解敌情就冲锋是赌博
- **为什么包围有效**：正面进攻靠的是个人能力——强的人能冲过去，弱的人卡在那里。包围靠的是体系——体系不依赖英雄，依赖的是让错误无处藏身。你不需要团队里每个人都是高手，你需要的是让系统的高手和普通人的产出差距缩小
- **反面案例**：某团队接到迁移任务——把单体服务拆成微服务。第一天就动手拆，两周后发现拆出来的微服务之间有循环依赖，又花了三周解循环依赖，最后发现还不如原来的单体。如果先花三天画出模块依赖图，就会发现有三个模块必须一起拆、两个模块不能先拆——先拆哪个后拆哪个有严格的顺序。先画图只需要三天，不画图就浪费了五周

> 135 提醒：拿到难题的第一反应不应该是"动手写"，而是"先做哪些准备能让动手写时不踩坑"。准备的投入越大，执行时的风险越小。

## 二、堵死回退路径——让技术债没有退路

毛泽东要求"切断敌之退路"——不让被围的敌人跑掉。跑了就要重新围，重新围就要付出更大的代价。在软件里，"跑掉"的问题就是回退——新方案上线后出问题，团队想回退到旧方案，但旧方案的接口已经被改了，数据格式已经变了，回退反而引发更大的问题。

- **没堵死回退路径的典型场景**：
  - 数据库 schema 改了，但没有写反向迁移脚本——想回退只能手工修数据，修一晚上还不一定对
  - 旧 API 被删了，但调用方还没全切过来——回退后调用方全部 404，下游服务级联失败
  - 配置格式换了，但没有兼容层——回退后服务起不来，连"回退到正常"本身都成了故障
  - 消息格式从 JSON 改成 Protobuf，消费端还没全更新——回退后新消息消费者不认识，消息队列堆积到磁盘满了
  - 发布了新版本的前端静态资源，但 CDN 缓存没刷新——回退后部分用户看到的是新旧混合的前端，页面完全错乱
- **堵死回退路径的做法**：
  - 每一次 schema 变更都写 forward + backward 两个迁移脚本，并在 staging 环境实际执行一遍回退验证——不是"写了回退脚本"就算安全，而是"在 staging 跑过回退"才算安全
  - 新旧 API 并行运行至少一个发布周期，监控旧 API 流量归零后再下线——不是"调用方说切完了"就安全，要看流量数据
  - 配置变更用 feature flag 控制，回退只需关 flag，不需要重新部署——回退时间从"小时级"缩短到"秒级"
  - 数据格式变更使用双写 + 双读过渡期，确认所有消费端都已切换后再停写旧格式——双写期间的磁盘和计算开销是保险费，不是浪费
  - 静态资源变更使用不可变文件名（hash fingerprint）+ 按版本号分发——每个版本的前端资源独立存在，回退只需切版本号，不存在缓存不一致
- **堵死回退路径的本质**：不是"不让回退"，而是"让回退成为一次安全的、可预期的操作而不是一场灾难"。最好的回退是按一个按钮，最差的回退是全员通宵。回退演练应该像消防演习一样定期做——不是出事了才第一次练

> 135 提醒：每一次重大变更之前问自己——如果这个变更上线后出问题，回退需要几分钟？如果答案是"不知道"或"要好几个小时"，说明回退路径没堵好，应该先解决这个问题再推进。

## 三、多团队协调——像各纵队协同包围

淮海战役不是一支部队打的，是华野、中野几十万部队协同作战。毛泽东在电报里明确规定了每个纵队的进攻方向和到达时间——没有这个协调，包围圈就会有缺口，敌人就会从缺口突围。包围的强度取决于最薄弱的那个方向。

- **协调失败的表现**：
  - 前端团队重构了 API 调用方式，但后端团队不知道——上线后前端全部报错，故障排查花了四个小时才发现是接口不匹配
  - 平台团队升级了消息队列版本，但业务团队还在用旧版 SDK——升级当天全线消息丢失，业务团队以为是自己的 bug 排查了一整天
  - 三个团队同时改同一个共享库——互相覆盖，谁都不知道线上跑的是哪个版本，回滚后又被另一个团队的 CI 覆盖回去
  - 运维团队改了环境变量命名规范，但没通知开发团队——新部署的服务读不到配置，全部启动失败
  - 数据团队改了下游数据表的分区策略，但依赖这张表的报表服务不知道——第二天早上报表全部超时，老板问数据为什么还没出来
- **协调成功的做法**：
  - 建立"包围面"文档——明确每个团队负责哪个方向、什么时间到位、接口契约是什么。这个文档不是 wiki 上的一篇长文，而是一张表：团队 / 负责范围 / 到位时间 / 验收标准
  - 用 contract test 代替口头约定——前端后端各写测试验证接口行为，谁违反契约谁的测试就挂。口头约定会在发布压力下被忘记，测试不会
  - 建立"进攻时间表"——不是所有团队同时动手，而是按顺序：基础设施团队先到位 → 平台团队升级 → 业务团队接入。并行不是效率，按序才是安全
  - 设一个"合围检查点"——所有团队各就各位后，跑一次端到端集成测试，确认包围圈无缺口。这个检查点必须在总攻前通过，不通过就推迟总攻
  - 指定一个"战役指挥员"——不是项目经理催进度，而是一个能做技术决策的人负责判断各团队是否到位、何时发起总攻。这个人必须有权限叫停总攻
  - 每个团队在自己的"包围面"完成后主动报备——不是等指挥员来问，而是主动发消息："我的部分已就绪，contract test 全绿，等待合围检查"

> 135 提醒：跨团队协调不是开一个 kick-off 会议发一个文档就完事——要像军事协同一样，明确每个团队的到位时间、到位标志、以及到位后如何确认。

## 四、不要急于进攻明显目标——先解决侧翼威胁

毛泽东在辽沈战役中坚持先打锦州而不是先打长春——林彪想打长春（近、容易），毛泽东坚持打锦州（远、难但关键）。因为打锦州可以关门，打长春只是啃一个据点。先打长春，锦州的守军可以跑，跑了就要重新打。先打锦州，长春就是囊中之物。

- **"先打长春"的技术决策**：
  - 团队看到用户最常抱怨的 UI 问题，把两周时间花在打磨界面上——但底层的数据一致性问题没解决，用户界面再漂亮数据也是错的
  - 新项目直接上手写业务逻辑，跳过架构设计——前两周产出很多，第三周开始被架构瓶颈拖住，后面的产出断崖式下降
  - 性能优化直接加缓存，不分析瓶颈在哪——缓存加了一层又一层，真正的瓶颈是数据库的一个全表扫描，缓存解决不了全表扫描
  - 遇到 legacy 系统迁移，先迁移最简单的模块"积累信心"——简单模块迁移完了，最难的核心模块反而因为前面的迁移引入了新的耦合变得更难迁
- **"先打锦州"的技术决策**：
  - 先花一周把日志和监控体系建好——虽然不产出功能，但后面每一个性能问题都能在三分钟内定位，省下的调试时间是十倍以上
  - 先把核心数据模型和接口契约定死——后面不管谁来做业务功能，都不会在数据层面产生冲突。数据模型是地基，地基不牢上面都白搭
  - 先解决共享库的版本管理和发布流程——虽然不直接产生业务价值，但消除了三个团队互相踩脚的根源。共享库的混乱是所有依赖团队效率的上限
  - 先搭建 CI/CD 流水线和自动化测试骨架——后面每个团队提交代码都有安全网，没有安全网的团队跑得越快摔得越惨
- **如何判断什么是"锦州"**：问自己一个问题——"如果这个模块搞定了，后面至少三个模块的开发会变简单吗？"如果是，那就是锦州。如果搞定了只影响一个模块，那可能只是长春。锦州的特征是"搞定它之后后面的路突然变宽"，长春的特征是"搞定它之后后面的路还是一样窄"

> 135 提醒：最有价值的技术工作往往不是最紧急的那个——先做"让后面所有工作变简单"的事，而不是"让用户最快看到"的事。

## 五、包围不等于拖延——有节奏地收缩

毛泽东的包围不是慢吞吞地围，而是有明确节奏的"围-紧-打"——围住后迅速收紧，不给敌人喘息时间。包围是手段，歼灭才是目的。如果围而不打，包围就变成了对峙，对峙消耗的是自己的士气和资源。

- **包围变成拖延的表现**：
  - "我们先做一个月的调研"——一个月后调研报告写完了，代码一行没写，业务方已经失去耐心
  - "我们把所有 edge case 都想清楚再动手"——想了两周，发现边写边想更快，前面想的大部分都白想了
  - "我们等另一个团队先完成他们那部分"——等了三个迭代，对方还没开始，你的团队已经养成等别人的习惯
  - "我们再招一个资深工程师来主导"——招人花了两个月，来了又要一个月熟悉代码，三个月过去了问题还在那里
- **有节奏的包围**：
  - spike 最多三天——三天后必须有结论：能做还是不能做、怎么做、大概多久。三天没有结论说明问题拆解不够，需要换一种方式分析
  - 包围阶段设硬性检查点——"本周五之前完成依赖分析和测试覆盖，下周一开始写代码"。检查点不是建议，是纪律
  - 每个包围动作都产出可验证的 artifact——依赖图文档、contract test、迁移脚本——不是"想清楚了"，而是"准备到位了"。想清楚是不可验证的，artifact 是可验证的
  - 包围阶段不超过一个迭代——如果一个迭代还没准备好，说明问题拆得不够细，需要把问题拆成更小的块再分别包围
  - 每个包围动作都要被 code review——包围阶段产出的测试、脚本、文档都要有人看过。没人看过的包围动作可能方向就错了，方向错了包围再严也没用
- **判断包围还是拖延的简单标准**：问自己"这个星期结束了，我能拿出什么具体的东西给别人看？"如果拿不出来，要么是拖延，要么是包围面太大了需要拆小

> 135 提醒：包围的每一步都要有可交付的产出。如果包围阶段结束拿不出任何 artifact，那不是包围，是拖延。

## 六、总攻时机——包围到位后的果断出手

毛泽东在包围完成后要求"不失时机地发起总攻"——包围是为了创造必胜条件，条件具备了就要果断打。犹豫会丧失战机，敌人可能利用犹豫时间重新组织防御。

- **总攻时机的判断**：
  - 所有依赖模块的接口已锁定并通过 contract test——不是"差不多定好了"，而是"测试全绿"
  - 回退路径已验证——灰度切回旧版本可以正常工作，且回退时间在预期范围内
  - 跨团队协同的检查点已通过——端到端集成测试全绿，每个团队都确认到位
  - 监控和告警已部署——总攻后如果有问题能立刻发现，不需要等用户投诉
  - 所有人都已通知——不只是技术团队，运维、客服、业务方都知道今天有重大变更，有问题能第一时间响应
- **果断出手的表现**：
  - 一次变更做透——不要"先改一半下个迭代再改另一半"，半改的代码是最危险的代码，它既不是新的也不是旧的
  - 全量灰度快速推进——如果灰度 1% 没问题，不要等一周才推到 10%，观察期够了就果断推。慢推不等于安全，慢推等于延长风险暴露时间
  - 变更后的第一个迭代全力监控——把主要精力放在观察和快速响应上，而不是立刻开始下一个功能。打赢了要打扫战场，不是立刻打下一场
- **总攻失败的处理**：如果总攻后发现严重问题，果断回退——不要"再修修看"。回退不是失败，在错误的方向上坚持才是失败。回退后复盘包围阶段的遗漏，修复遗漏后重新包围、重新总攻。每一次失败都让下一次的包围更严密
- **什么情况下不要发起总攻**：如果有任何一个前置条件没满足，就不要发起。宁可多等一个迭代把包围做扎实，也不要带着缺口冲锋。推迟总攻的代价是时间，带着缺口冲锋的代价是信心——团队的信心比一个迭代的时间更宝贵

> 135 提醒：包围的目的是让总攻变成一件确定的事。如果包围到位了还不敢出手，说明包围没到位；如果没包围就出手了，说明你在赌。

## 结语

毛泽东"围而歼之"的核心方法论：先建立必胜条件再动手，让敌人无处可退，让各部队协同到位。

给今天技术团队的一句落地建议：**每一个重大技术变更，都先回答三个问题——（1）依赖关系是否全部摸清并锁定？（2）回退路径是否验证过可在五分钟内完成？（3）涉及的团队是否都在"包围面"文档中明确了到位时间和验收标准？——三个"是"才能发起总攻。**
