# 从《团结一切抗日力量，反对反共顽固派》看软件团队协作：很多团队不是被一次烂 PR 或一次故障击垮，而是在几次受伤后就开始扩大化，把某个同事、某个组或某套技术栈整体判死，结果把本来还能合作、还能修复、还能升级的力量也一起推开了

日期：2026-04-21

原文对应：
《团结一切抗日力量，反对反共顽固派》

写作背景：
1940年2月，抗日战争进入更加复杂的阶段。一方面，确实存在破坏合作、制造摩擦、打击进步力量的顽固派；另一方面，又不能因为这些破坏力量的存在，就把一切仍可争取、仍可联合、仍可合作的对象一概打成一类。毛泽东在《团结一切抗日力量，反对反共顽固派》里反复强调的，是一种很强的治理方法：要分类判断，不要一刷子抹平；要同时执行斗争政策和团结政策，而不是只剩一种动作；要靠可验证的事实建立公信力，而不是靠空口号；还要警惕借“统一”之名行压制、吞并、扩权之实的假统一。这套方法放到今天的软件开发、团队协作、工程治理、跨组合作和技术栈整合里，依然非常有现实感。

说明：
今天把这篇文章转译到软件开发语境里，当然不能照搬战争年代的政治表达。但它里面那种“看见问题对象，同时仍然保留分类能力”“不能因为一次烂 PR、一次故障、一次 review 冲突，就把一个人、一个组、一套技术栈乃至一类工程方法整个判死”“既要处理真正破坏工程秩序的人和事，也要继续团结和经营仍然有价值的协作力量”“工程信用要靠可验证事实建立”“统一不能变成某一方借规范、平台化、栈统一之名压平差异和扩张控制”的方法，对今天的软件团队特别重要。很多工程团队后来越做越僵、越做越冷，不是因为团队里真的只剩烂代码和烂人，而是因为大家在连续几次受伤后，慢慢把“这次 PR 很差”“这个组这次协作失真”“这套技术栈这次出了事故”升级成了“这个人整体不行”“那个组都不靠谱”“这套栈整体有毒”。这样一来，被关闭的就不只是坏对象，还包括那些本来还能合作、还能被带起来、还能帮助团队升级的大量中间力量。

核心思想的当代转译：
如果把《团结一切抗日力量，反对反共顽固派》翻成今天软件团队协作的语言，我会把它概括成一句话：
很多团队不是被一次烂 PR 或一次故障击垮，而是在几次受伤后就开始扩大化，把某个同事、某个组或某套技术栈整体判死，结果把本来还能合作、还能修复、还能升级的力量也一起推开了。真正成熟的工程治理，不是看不见风险，也不是对所有人一味讲和，而是在问题面前仍然保留分类、分层、分政策的能力：对真正破坏工程秩序的人和事坚决处理，对大多数仍然可合作、可改进、可成长的对象继续投入，对工程信用用一组可核验的事实来建立，同时识别和阻断那些借“统一架构”“统一规范”“统一技术栈”之名消灭差异、夺走判断权的假整合。

## 一、第一层提醒：很多团队后来越来越难合作，不是因为真的谁都不行，而是在几次受伤以后，把少数坏样本扩大成了对整个人群和整套系统的判决

软件团队当然会受伤。

这很正常。

你会遇到：
- 某个 PR 写得很乱，review 一圈下来几乎得重做
- 某次线上故障明明可以避免，却因为低级失误炸了
- 某个跨组接口答应得很好，最后交付却严重缩水
- 某位同事在设计评审里态度很硬，让人很不舒服
- 某个模块的 owner 总在关键时刻掉链子
- 某套技术栈在你们场景下踩了一个很大的坑

遇到这些事，
工程师生气、失望、想拉黑、想再也不合作，
都很正常。

问题不在这些情绪本身。

问题在于，
很多团队一旦连续被伤几次，
会在情绪最强的时候完成一个特别危险的动作：
把“这一次问题出在这个对象身上”升级成“这个对象整体都不行”，
再进一步升级成“这一类对象都不值得再信”。

于是团队里就会慢慢冒出这些很熟悉的话：
- 这个人整体水平不行
- 那个组做什么都不靠谱
- 这套栈就是有毒
- 产品侧都不懂工程
- 测试只会卡人
- 后端都不关心体验
- 前端都喜欢搞花活
- 外包都不能信

这些判断之所以危险，
不是因为它们完全没有现实刺激。

恰恰相反，
它们通常都来自很具体、很真实、很让人生气的经历。

所以说这些话的人，
会觉得自己不是偏激，
而是“吃过亏，见过真相”。

可真正的问题就在这里。

你遇到过一次烂 PR，
不等于这个人所有工作都烂。

你碰到过一次跨组失真，
不等于那个组从此没有任何合作价值。

你踩过一套技术栈的坑，
不等于那套栈在所有场景里都该被一键判死。

一旦团队开始用扩大化方式判断人、组和技术选择，
后面的动作就会变形：
- 本来该处理一次具体失误，变成对某人长期贴标签
- 本来该修复一次协作断裂，变成对某组整体失去合作意愿
- 本来该复盘一次技术使用错误，变成对整套技术方案直接封杀
- 本来该把问题聚焦在流程和证据上，变成让情绪替代工程判断

最后团队会进入一种特别吊诡的状态：
明明还存在大量能写好代码、能扛主线、能把问题修正过来的同事和伙伴，
大家却越来越像一群彼此预设不信任的人。

而这往往不是因为团队里只剩坏对象，
而是因为你先把团队看粗了。

所以这篇文章给软件人的第一个提醒就是：
不要把受伤经验直接升级成你的工程哲学。

坏对象当然要识别，
但坏对象的存在，
不应该成为你放弃分类能力的理由。

很多团队后来越来越难合作，
不是因为没人值得合作，
而是因为在几次受伤以后，
亲手把本来还能联合、还能修复、还能升级的力量也一起判掉了。

## 二、第二层提醒：工程治理真正难的地方，不是“态度硬一点”，而是始终保持分类、分层、分政策

毛泽东在这篇文章里一个特别强的方法，
就是对不同部分的人，
采取不同政策。

这放到软件团队里，
几乎可以直接翻译成一句话：
工程治理不是统一发火，
也不是统一讲和，
而是持续分类、分层、分政策。

很多团队为什么越做越粗？

因为最省事的方式，
就是一刀切。

于是你会看到这些特别常见的反应：
- 某个新人连续两个 PR 写得不好，团队立刻默认“这人不适合写核心代码”
- 某个服务组一次协作失真，其他组就直接把它列入“高风险合作对象”
- 某次线上事故发生在某个模块，团队就默认“这个模块没人能救，只能重写”
- 某套技术栈在一类场景下踩坑，大家就喊着“以后全部禁用”
- 某位 reviewer 的表达方式太冲，团队就把他的所有技术意见一并降权

表面上看，
这种一刀切像是在提高效率。

可实际上，
它会把本来复杂的工程现场，
压成一个粗暴的标签系统。

而一旦进入标签系统，
真正会发生三件事。

第一件事，
真正有问题的点和本来还能修复的人被混在一起处理。

第二件事，
团队会迅速失去公平感。

第三件事，
管理动作会越来越容易伤到那些本来还值得投入、值得保护、值得争取的人。

所以成熟的工程治理，
不是把对象只分成“靠谱”和“不靠谱”两类，
而是至少要学会更细一点地看。

比如一个同事连续交出低质量 PR，
你至少可以分清几层：
- 是能力明显不足、又拒绝反馈、持续破坏规则的人，这种要严肃处理
- 是基础薄弱但愿意学习、需要更强 review 和 mentor 支持的人，这种要重点带
- 是上下文不足、任务定义模糊导致写差的人，这种重点不在怪人，而在补上下文和任务拆解
- 是当下状态差、角色错配、任务超出能力带宽的人，这种更需要调整位置和节奏

跨组合作也一样。

不能只分成“好合作的组”和“烂组”。

更稳的分法，
往往是：
- 明显长期不守约、反复失真、持续制造高风险接口的对象，要设硬边界
- 有摩擦但可修复、问题主要在机制和节奏上的对象，要重做接口规则
- 愿意协作、只是信息不同步的对象，要继续经营合作链路
- 真正能共同扛主线的对象，要升级成主干同盟

技术栈也一样。

有的是在当前场景不适合，
有的是被错误使用，
有的是成熟但需要治理配套，
有的是确实不值得再继续投入。

如果你把它们全部混成“先进”或“有毒”，
你就很难知道自己到底应该修哪一层问题。

所以真正成熟的工程团队，
从来不是一句“以后这类全部不准”，
而是始终保留判断：
- 这个问题属于人、流程、上下文还是技术本身
- 这是一次失误、阶段性错配，还是长期不可接受的问题
- 哪些要坚决处理
- 哪些要继续教育和带
- 哪些要保护和扶正
- 哪些要重新设计合作方式

很多团队不是没有制度，
而是制度背后没有分类能力。

于是制度执行到最后，
就会从工程治理滑成情绪治理，
从质量判断滑成标签打击。

而一刀切最大的代价，
不是显得不够“人性化”，
而是会把真正有价值的人、合作面和技术机会一起打伤。

## 三、第三层提醒：工程里最稳的做法，不是只会抓问题，也不是只会讲协作，而是该处理的处理，该团结的团结

这篇文章特别厉害的地方，
在于它不是只讲一个动作。

它讲的是两条政策并行。

一方面反对和打击真正破坏合作、破坏秩序、破坏整体的人和事；
另一方面又尽可能联合仍然可合作、可成长、可争取的力量。

翻到今天软件团队里，
这是一条非常硬的方法：
成熟的工程治理，
不是只会卡，
也不是只会哄，
而是同时具备两种能力。

很多团队会掉进两个极端。

第一个极端是：
只会抓问题。

比如：
- 只会在 review 里指出问题，不会补上下文
- 只会在复盘里说哪里错，不会设计下一次怎么不再错
- 只会强调红线，不会扶正那些本来愿意进步的人
- 只会记录事故，不会经营协作关系
- 只会批评跨组不配合，不会主动建立更好的接口机制

这种团队短期看起来很硬，
但时间一长，
问题就会出来：
- 大家越来越怕暴露真实问题
- 新人越来越不敢提问
- review 变成防御场，而不是共同提升场
- 跨组关系越来越僵，信息流越来越少
- 真正有责任心的人会逐渐疲惫，因为付出常常换来的是更重的指责压力

第二个极端是：
只会讲协作和包容。

比如：
- 明明显著低质量的 PR 也先过了，怕伤人
- 明明接口承诺一再失真，也不敢压责任，怕影响关系
- 明明技术风险已经很大，也只说“大家互相理解一下”
- 明明某个 owner 长期失职，也始终不调整，怕团队震荡
- 明明规范已经被踩穿，也总说“先把功能做出来”

这种团队短期看起来和气，
但最后往往会被少数问题对象拖着走：
- 守标准的人越来越吃亏
- 会拖延、会甩锅的人越来越占便宜
- review 质量持续下滑
- 系统债越积越厚
- 团队对“底线”这个词逐渐失去信任

所以成熟的工程治理从来不是二选一，
而是两条政策并行。

对真正破坏质量和协作秩序的对象，
就是一边守住红线、明确后果、压实责任。

对大多数仍可合作、可教育、可成长的人，
就是一边继续投入 mentor、上下文、反馈和成长空间。

对跨组合作来说，
就是一边阻断反复失真的接口行为，
一边继续经营那些能共同把主线扛起来的合作关系。

对技术栈来说，
就是一边剔除明显不适合当前场景的选择，
一边继续保留对差异方案、边界场景和更优工程路径的研究空间。

很多团队最后特别僵，
不是因为不知道要守底线，
也不是因为不知道要合作，
而是永远只会用一种方式处理所有问题。

可真实工程现场里，
坏的对象要处理，
中的对象要带，
好的对象要保护，
核心的对象要扶正和放大。

这才是软件团队真正的治理厚度。

## 四、第四层提醒：工程信用不是靠口号建立的，而是靠一组任何人都能核验的事实建立的

这篇文章里还有一个特别值得软件团队学的地方，
就是“十没有”式的事实表达。

它厉害的地方在于：
不是用大词证明自己正确，
而是用别人可以核验的事实证明自己可信。

这件事放到工程治理里，
太重要了。

因为今天很多团队最容易出的问题，
就是嘴上特别重视质量，
现场却没有事实底盘。

比如我们常听到：
- 我们团队代码质量很高
- 我们很重视工程规范
- 我们上线流程很成熟
- 我们协作氛围很好
- 我们很重视稳定性

这些话不是不能说，
但它们有一个共同问题：
太像自我评价，
不像事实。

而真正能建立团队信用的，
从来不是形容词，
而是事实清单。

一个软件团队如果真想让组内、跨组、管理层甚至候选人相信自己，
最好能拿出一组像“十没有”或“十有”那样，
任何人都能去 dashboard、流水线、RCA 库、PR 系统里核验的事实。

比如你可以强迫团队回答：
我们的工程信用，能不能稳定地由这些事实支撑？

例如：
- 核心服务过去12个月 P0/P1 事故的真实数量是多少
- 核心仓库 PR 平均 review 时长和返工率是多少
- 主链路服务的 test coverage、CI pass rate、flaky 比例分别是多少
- 线上事件发生后 48 小时内 RCA 完成率是多少
- 重要服务是否都有明确 owner、oncall 和值班机制
- 发布 checklist 的真实执行率是多少
- 高优先级技术债关闭率和新增率分别是多少
- 告警噪音占比和有效告警比例是否在改善
- 新人 30/60/90 天是否有明确交付与辅导记录
- 跨组接口 SLA、延期率和变更同步质量是否可核验

你会发现，
一个团队到底有没有工程文化，
很多时候不是看它的 README 写得多漂亮，
而是看这些事实能不能拿出来。

事实清单至少有四个作用。

第一个作用，
减少空话。

一旦你被迫把“质量很好”翻译成事实，
很多原本靠氛围撑着的自我感觉会瞬间见光。

第二个作用，
帮助团队区分“个别人有问题”和“系统真的有问题”。

如果坏结果只集中在少数对象，
那就别急着把整个团队讲成不行；
如果关键指标在系统层面持续恶化，
那才说明问题已经上升到治理层。

第三个作用，
它会让复盘真正有抓手。

很多团队复盘总是停在“沟通再加强”“以后注意点”“大家辛苦了”这种话，
本质上就是因为平时没有事实底稿。

第四个作用，
它能在冲突时帮你守住判断权。

因为一旦跨组矛盾和上线压力上来，
没有事实底盘的团队最容易被声音大的人带走。

可如果你有一组可持续追踪的事实，
你就更容易知道：
现在是在被情绪攻击，
还是系统真的有问题。

所以对软件团队来说，
“十没有”的现代转译，
不是写一份更好看的宣传文案，
而是建立自己的工程事实底盘。

很多团队之所以长期争论很多、共识很少，
不是因为大家智商不够，
而是长期把判断建在印象上，
没有建在证据上。

而没有事实支撑的工程信用，
到了关键时刻，
通常一碰就散。

## 五、第五层提醒：软件团队里的“统一规范”“统一平台”“统一栈”也要警惕假整合；很多表面上的统一，本质上是在借治理之名扩权和压平差异

毛泽东在文里对“假统一”的识别非常锋利。

它提醒人们，
不是所有打着统一、合作、整合名义的动作，
都真的在服务共同目标。

有些所谓统一，
其实是借统一之名，
行专断、吞并、压制、扩权之实。

这放到今天软件团队里，
就是那些特别常见的“假整合”。

它们表面上讲的是：
- 统一技术栈
- 统一规范
- 统一平台
- 统一中台
- 统一部署方式
- 统一工程流程

这些词听起来都很合理，
甚至听起来很先进。

可实际推进下去之后，
团队往往会慢慢发现：
- 某一方的判断权和话语权明显变大了
- 另一方的场景差异和真实约束被当成麻烦
- 原本合理存在的不同做法被粗暴抹平
- 一线团队承担更多迁移成本，却没有真正获得更好的支持
- “统一”最后只是让一方更好管，而不是让系统更好做

于是团队里会出现一种特别熟悉的感觉：
名义上更一致了，
实际上更僵了；
名义上更规范了，
实际上更多人只是表面配合；
名义上更现代化了，
实际上工程判断空间更窄了。

现实里常见的假整合很多。

比如：
- 推一个统一平台，口头上说为了降本提效，最后变成所有业务被迫迁就平台团队的节奏
- 推一个统一技术栈，口头上说为了维护一致性，最后却把适合特殊场景的差异选择全部消灭
- 推统一代码规范，口头上说为了质量，最后实质上是某个强势团队把自己的工作习惯变成全公司的政治正确
- 推统一中台，口头上说为了复用，最后一线团队反而被抽走了自主修复能力
- 推统一流程，口头上说为了可控，最后复杂性只是从系统内部搬到了沟通和等待里

所以团队在做任何“统一”之前，
都应该强迫自己回答几个问题。

第一，
这次统一到底有没有真正共同目标？

是为了减少重复建设、降低运维复杂度、提升稳定性、改善协作效率，
还是只是为了让某一方更好管理、更好控盘？

第二，
这次统一是否保留了必要差异？

真正成熟的工程治理，
不是消灭所有不同。

不同业务阶段、不同性能要求、不同团队能力结构，
本来就可能需要不同解。

如果统一的真实目标，
是把所有差异都压平，
那它通常已经开始偏向假整合了。

第三，
统一后的收益和成本是不是多方共享？

如果收益主要流向平台方、规范制定方或强势团队，
而迁移成本、适配成本、迭代变慢的代价全由一线承担，
那这就不是共同升级，
而更像是资源再分配。

第四，
被统一的一方有没有真正参与规则形成？

很多假整合的问题，
不是方案一点价值都没有，
而是它根本没有把被整合对象当成共同建设者，
只把他们当执行对象。

这时即使表面上都“统一”了，
底下的人心也不会真的统一。

他们会表面配合，
私下绕过；
表面同意，
心里不服；
表面迁移，
实际消极执行。

最后推动者会抱怨：
为什么统一以后反而更低效？

答案往往很简单。

因为你做的不是整合，
而是吞并。

所以这篇文章给软件团队一个非常现实的提醒：
统一不是天然正确的，
平台化也不是天然高级的。

任何打着统一和治理旗号的动作，
都必须接受审查：
它到底是在服务共同目标，
还是只是在放大某一方的权力、压缩别人的判断空间。

如果是后者，
它就算短期压过去了，
长期也会把工程主动性、协作意愿和真实效率一起做坏。

## 六、第六层提醒：把文章真正落到软件团队动作上，关键不是“懂得别扩大化”，而是把分类治理、事实立信和假整合审查看成同一套工程系统

很多文章读完以后，
最容易出现的情况是：
道理都懂，
回到日常协作还是老样子。

所以如果真要把这篇文章落到软件团队动作上，
最重要的不是感叹“分类很重要”，
而是把它翻成一整套能反复运行的工程动作。

我更建议团队至少做六件事。

第一件事，
画出你的协作对象地图。

不要再把问题都混在“开发同学”“别的组”“这套栈”这种大词里。

至少把人、组、模块、接口、技术栈、平台方、规范制定方分别拆开看。

只有对象拆开，
后面才可能分层。

第二件事，
为每次冲突和事故做归因分层。

至少分清：
- 是个人能力问题
- 是上下文和任务设计问题
- 是流程缺口问题
- 是协作接口问题
- 是技术栈边界问题
- 是治理方式本身的问题

不把这些拆开，
你最后只会得到一句空话：
“这人不行”或者“这套栈不行”。

第三件事，
为不同层写不同政策。

不要一出问题就只剩“以后都不让这样”或“先算了”两种动作。

更有效的做法是：
- 明确破坏规则的对象怎么处理
- 愿意成长但能力薄弱的对象怎么带
- 跨组合作失真怎么重做接口机制
- 技术栈不适配怎么设边界
- 强对象和主干力量怎么保护和放大

第四件事，
给团队建立一张“工程事实清单”。

不要再把团队信用建在口号上。

你要写清楚：
- 哪些指标能证明我们工程上值得信
- 这些指标谁维护
- 多久复盘一次
- 变好还是变坏的阈值是什么
- 一旦恶化，谁来推动动作

第五件事，
把所有统一动作放进“真假整合审查”里。

至少审四件事：
- 有没有真正共同目标
- 有没有保留必要差异
- 收益和成本是不是多方共享
- 被统一的一方有没有参与规则形成

如果这些问题大部分都答不上来，
那这次统一大概率还不够成熟。

第六件事，
给自己的工程治理建立复盘机制。

因为分类不是一次性的。

人会变，
组会变，
系统会变，
技术栈的适配条件也会变。

今天需要坚决限制的对象，
半年后也许已经被带起来；
今天还值得保留研究空间的技术方案，
下个阶段也可能真的不适合了；
今天看似稳定的合作关系，
在压力场里也可能变成高风险接口。

如果没有复盘，
分类迟早会重新塌回标签判断。

所以软件团队真正该做的，
不是学会几个口号，
而是把分类治理、事实立信和假整合审查，
做成自己长期协作和工程判断的一套系统。

只有这样，
团队才不会在几次受伤之后，
一会儿过度发火，
一会儿过度讲和，
一会儿全面封杀，
一会儿又被“统一大义”带着走。

## 给自己的三个自检问题

1. 我最近是不是因为几个烂 PR、几次协作失真或一次故障，就把某个同事、某个组或某套技术栈整体判死了？这种扩大化，是否已经在关闭本来还能合作和修复的空间？
2. 我们团队现在处理问题时，是不是其实只剩一套办法？哪些对象本来应该分类、分层、分政策，却被我们一刀切地贴标签、拉黑或统一施压了？
3. 我们口中的工程质量、协作文化和统一治理，到底有没有一组可核验的事实和真正共享的目标？还是很多信用只是口号，很多统一只是借规范、平台化、架构收口之名扩权和压平差异？
