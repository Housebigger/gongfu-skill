# gongfu-skill 设计规范（v1.0）

日期：2026-06-15
性质：本文档从第一批 3 个实战 skill（industry-scan / startup-feasibility / problem-diagnosis）的制作实践中反推提炼，是后续 skill 制作的标准。

---

## 一、什么是 gongfu-skill 里的「skill」

skill 是把仓库里的文档知识**蒸馏成可调用的判断服务**。

| 不是什么 | 是什么 |
|---|---|
| 不是又一篇文档 | 是一个「输入→判断→输出」的服务 |
| 不是给人读的文章 | 是给 AI agent 加载后执行的过程 |
| 不是百科全书条目 | 是一个解决具体问题的专业判断工具 |

一句话：**文档回答「这是什么」，skill 回答「我这个情况该怎么办」。**

---

## 二、skill 的标准结构

每个 skill 是一个目录，包含一个 `SKILL.md`：

```
skills/<skill-name>/SKILL.md
```

### SKILL.md 的必须组成

通过第一批 3 个 skill 的实践，沉淀出以下标准结构：

#### 1 · Frontmatter（元信息）

```yaml
---
name: skill-name              # 小写连字符，≤64字符
description: "Use when <触发条件>. <一句话行为>."  # ≤1024字符
version: 1.0.0
author: gongfu-skill
license: MIT
metadata:
  gongfu_skill:
    phase: 3                  # 第三阶段产出
    source_layers: [...]      # 蒸馏自哪些层
    methodology: [...]        # 用了哪些方法论工具
---
```

**description 的写法**：必须以「Use when ...」开头，说明什么时候调用这个 skill。这是 AI agent 匹配用户意图的依据。

#### 2 · 概述（Overview）

2—3 段话：这个 skill 是什么、解决什么问题、蒸馏自哪些内容。

**关键**：必须明确写出「它解决的问题是 ___」——一句话说清楚这个 skill 的价值主张。

#### 3 · 什么时候用 / 不要用于（When to Use）

- 正向触发条件（bullet 列表）
- **反向排除条件**（「不要用于：___」）—— 这是防止 skill 被误用的重要设计

#### 4 · 输入规格（Input）

表格形式列出：
- 哪些是必填、哪些选填
- 每个参数的含义和示例
- **颗粒度原则**：输入要简单到用户三句话能说清楚，不要让用户填一张表

#### 5 · 执行逻辑（Processing）

这是 skill 的核心——**分步骤的判断流程**。每个步骤：
- 有明确的输入（上一步的输出）
- 有明确的判断依据（引用源文档的具体内容）
- 有明确的输出（传递给下一步）

**从实践中发现的规律**：
- 步骤数 4—6 步为宜（industry-scan 是 4 步，startup-feasibility 是 5 步，problem-diagnosis 是 5 步）
- 第一步通常是「归类/还原」（把用户输入映射到 skill 的知识框架）
- 最后一步通常是「方法论校验/红线提醒」（保证诚实性）
- startup-feasibility 发现了一个重要模式：**劝退检查应该在路径推荐之前**——不是所有评估都应该鼓励前进

#### 6 · 输出规格（Output）

给出输出模板，包含：
- 判断结果（信号/可行性/诊断）
- 行动建议（可操作的，不是「努力」）
- 风险红线（每个 skill 都有）
- 一句话总结

#### 7 · 测试用例（Test Cases）

**每个 skill 必须有至少 3 个测试用例**：
- 一个「典型成功」用例（输入命中正常路径）
- 一个「边界/劝退」用例（输入触发特殊处理，如劝退/不推荐）
- 一个「复杂/模糊」用例（输入信息不全或跨多个领域）

每个用例包含：输入 → 期望信号/要点 → 验证标准

**测试用例是 skill「可验证性」的保证**——没有测试用例的 skill 不算完成。

#### 8 · 源文档映射（Source Mapping）

表格列出本 skill 的每个判断依据来自哪个源文档。**这保证 skill 不是凭空捏造，而是可追溯的蒸馏。**

#### 9 · 诚实的边界（Boundaries）

每个 skill 必须有「诚实的边界」一节：
- 它不提供什么（投资建议/法律意见/医疗诊断等）
- 它的判断基于什么时间点的信息
- 个体差异提示

## 三、skill 的五项质量标准

从 3 个 skill 的制作中提炼的验收标准：

| 标准 | 含义 | 怎么检验 |
|---|---|---|
| **可调用** | AI agent 能根据用户意图匹配到这个 skill | description 的「Use when」是否准确描述触发条件 |
| **可验证** | 有测试用例能跑通 | 每个测试用例的输入→输出是否明确、可复现 |
| **可组合** | 多个 skill 能串联 | skill 的输出能否成为另一个 skill 的输入（如 problem-diagnosis → industry-scan → startup-feasibility） |
| **可追溯** | 判断有据 | 源文档映射是否完整——每个判断都能追溯到具体文档 |
| **诚实** | 知道自己不能干什么 | 「不要用于」+「诚实的边界」是否覆盖了误用风险 |

**不满足这五项的 skill 不算合格，需要补强。**

## 四、skill 之间的组合模式

第一批 3 个 skill 已经展现了组合潜力：

```
用户：「我30岁做Web开发，被AI威胁想转行，但又怕」
  │
  ▼
problem-diagnosis（诊断主要矛盾）
  │ 输出：主要矛盾=技能贬值vs收入需求；方向=升行不转行
  │
  ▼
industry-scan（扫描目标行业）
  │ 输入：行业=AI/数字经济，岗位=程序员，地域=用户所在地
  │ 输出：【转】信号 + 升级路线
  │
  ▼
startup-feasibility（如果用户还想创业）
  │ 输入：技能=编程+行业理解，财务/家庭/时间
  │ 输出：适不适合创业 + 推荐路径
```

这不是硬编码的 pipeline，而是 AI agent 根据用户意图自然编排。**好的 skill 设计时就考虑了它的输出能否被下一个 skill 使用。**

---

## 五、命名与分类约定

- 目录名：英文小写连字符（`industry-scan` / `startup-feasibility` / `problem-diagnosis`）
- 每个目录只含一个 `SKILL.md`
- 未来如需辅助文件（如更细的数据表/案例库），放在同目录下的 `references/` 或 `data/` 子目录

---

## 六、本规范自身的演进规则

本规范不是教条，是从实践中提炼的——它应该随着更多 skill 的制作持续更新。

- 每制作 3—5 个新 skill 后，回头审视本规范是否需要修订
- 如果实践发现规范有阻碍，改规范而不是削足适履
- 规范的修订同样遵循「先干再总结」——先发现模式，再固化规则

**方法论根基**：《改造我们的学习》（060）反对教条，实事求是——规范从实践来，不从理论来。
