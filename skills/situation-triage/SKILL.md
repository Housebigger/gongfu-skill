# 情况分诊 skill（路由层）

## 概述

这个 skill 是共富参谋插件的**路由层**。它不直接给判断，而是分析用户的自由文本描述，判断该路由到哪些下游 skill，并提取结构化信息传给它们。

它解决的问题是：**用户不应该需要知道「该用哪个 skill」——他只需要说出自己的情况，这个 skill 帮他分诊。**

就像医院的分诊台——你不知道该挂什么科，分诊护士听完你的描述，告诉你该去哪个科。

## 什么时候用

**总是先用这个 skill。** 它是共富参谋的总入口，所有用户输入都先经过它。

**例外**：如果用户明确指定了要什么（如「帮我评估一下创业」），可以直接跳到对应 skill。

## 输入规格

| 参数 | 必填 | 说明 |
|---|---|---|
| 用户原话 | 是 | 用户用大白话描述的情况（原样传入，不做预处理） |

## 执行逻辑

### 第 1 步 · 意图识别

从用户的自由文本中，识别意图类型。一段话可能包含多个意图：

| 意图信号 | 关键词/语义特征 | 路由到 |
|---|---|---|
| 困境/迷茫 | 「累/纠结/不知道/迷茫/该怎么办/纠结/怕」 | problem-diagnosis |
| 行业判断 | 行业名+「前景/行不行/方向/好不好」 | industry-scan |
| 创业意向 | 「创业/副业/自己做/开店/搞个/兼职赚」 | startup-feasibility |
| 成长需求 | 「学什么/考证/转行/提升/规划/怎么学」 | growth-planner |
| 协作需求 | 「合伙/合作/找人/团队/一起干」 | collaboration-match |
| 趋势前瞻 | 「未来/趋势/几年后/前景/会不会被替代」 | opportunity-radar |

**复合问题的处理**：如果识别到多个意图，按以下优先级组合：

```
优先级 1：problem-diagnosis（如果用户情绪明显低落/困惑，先诊断）
优先级 2：industry-scan（大多数情况需要行业背景）
优先级 3：根据其他意图追加（startup/growth/collaboration/radar）
```

### 第 2 步 · 信息提取

从自由文本中提取结构化信息（提取不到的字段标记为「未提供」）：

| 字段 | 提取方式 | 映射规则 |
|---|---|---|
| 行业 | 从描述中识别行业关键词 | 映射到 A—P 集群 |
| 岗位 | 从描述中识别岗位 | 保留原话 |
| 地域 | 从描述中识别省市/区域 | 映射到五大区域 |
| 年龄段 | 从描述中识别年龄或推断 | 20-25/28-38/38-50/其他 |
| 财务状况 | 从描述中识别财务信号 | 有结余/月光/负债/未提 |
| 家庭状况 | 从描述中识别家庭信号 | 支持/反对/单身/未提 |
| 情绪状态 | 从描述中识别情绪信号 | 正常/低落/焦虑/耗竭 |

### 第 3 步 · 路由决策

输出路由结果：

```
路由到：[skill1, skill2, ...]（按优先级排序）
提取到的信息：{industry, role, region, age, finances, family, emotional_state}
```

### 第 4 步 · 特殊处理

某些情况需要特殊处理，不走常规路由：

| 特殊信号 | 处理 |
|---|---|
| 情绪=耗竭（极度疲惫/心态崩了） | **先处理情绪**：建议先休息恢复，不急于给职业判断。用 problem-diagnosis 的「相持阶段」框架 |
| 涉及健康危机/自伤倾向 | **不路由到任何职业 skill**，建议寻求专业帮助 |
| 完全没有可识别的意图 | 主动追问：「你想了解哪方面？行业前景/创业/学习成长/还是遇到了什么困难？」 |
| 信息严重不足 | 补充追问关键字段（行业/岗位/地域） |

## 输出规格

```
## 分诊结果

### 识别到的意图
- 主意图：___
- 次意图：___

### 路由到
1. ___（skill名）
2. ___（skill名，如有）

### 提取到的信息
行业：___（集群_）  岗位：___  地域：___（区域_）
年龄：___  财务：___  家庭：___  情绪：___

### 特殊处理（如有）
（情绪耗竭/信息不足/健康危机等）

### 建议的 skill 执行顺序
先执行___，再执行___
```

## 测试用例

### 用例 1：复合问题（嵌入式工程师案例）

输入：「我30岁做嵌入式开发，同事都走了，很累，不知道该不该换」

分诊结果：
- 主意图：困境/迷茫（「累/不知道该不该换」）
- 次意图：行业判断（嵌入式开发的行业前景）
- 路由：problem-diagnosis → industry-scan
- 信息：行业=嵌入式/机器人(H)，岗位=嵌入式工程师，年龄=30，情绪=低落/耗竭
- 特殊处理：情绪=耗竭，建议先处理情绪再给职业判断
- 顺序：先 problem-diagnosis（处理情绪+诊断矛盾），再 industry-scan（行业前景）

### 用例 2：明确创业意向

输入：「我想在县城开个养老服务机构，不知道行不行」

分诊结果：
- 主意图：创业意向
- 次意图：行业判断
- 路由：startup-feasibility → industry-scan
- 信息：行业=养老(E)，地域=县城(⑤)，意向=创业
- 顺序：先 startup-feasibility（评估可行性），再 industry-scan（补充行业信号）

### 用例 3：模糊请求

输入：「我就是想看看有什么机会」

分诊结果：
- 意图：不明确（没有具体方向）
- 处理：追问——「你是想看你当前行业的前景，还是想找新的机会方向？你目前在做什么？」
- 不急于路由，先补充信息

### 用例 4：情绪危机

输入：「太累了不想干了什么都无所谓了」

分诊结果：
- 意图：情绪危机信号
- 特殊处理：**不路由到职业 skill**。建议休息+寻求支持。如果涉及自伤倾向，建议拨打心理援助热线。
- 不做职业判断——状态不对时做的决策不可靠。

## 源文档映射

本 skill 是路由层，不直接引用 strategy 层内容。它的判断逻辑来自：
- 7 个下游 skill 的输入规格（决定提取什么信息）
- 7 个下游 skill 的触发条件（决定路由到哪）
- problem-diagnosis 的情绪/阶段判断（决定特殊处理）

## 诚实的边界

1. 本 skill 是**路由器**，不直接给职业判断——它的价值是帮用户选对下游 skill。
2. 路由不可能 100% 准确——自然语言意图识别有误差。允许路由到多个 skill 而非强行选一个。
3. **情绪危机处理优先于职业建议**——如果检测到耗竭/危机信号，先处理人，再处理事。
4. 本 skill 不替代心理咨询/危机干预。
5. 信息提取不到时宁可追问，不要瞎猜——错误的信息会导致下游 skill 判断跑偏。
